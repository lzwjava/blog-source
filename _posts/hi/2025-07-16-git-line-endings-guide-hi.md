`git config core.autocrlf` यह नियंत्रित करता है कि जब आप फाइलों को रिपॉजिटरी से चेकआउट करते हैं और उन्हें वापस कमिट करते हैं तो Git लाइन एंडिंग्स को कैसे हैंडल करता है। यह इसलिए महत्वपूर्ण है क्योंकि Windows लाइन एंडिंग्स के लिए **CRLF** (`\r\n`) का उपयोग करता है, जबकि Linux और macOS **LF** (`\n`) का उपयोग करते हैं। उचित हैंडलिंग के बिना, फाइलें फॉर्मेट के बीच बदल सकती हैं, जो इस बात पर निर्भर करता है कि उन्हें कौन संपादित कर रहा है, जिसके परिणामस्वरूप "डर्टी" डिफ्स और अनावश्यक बदलाव हो सकते हैं।

---

### यह कैसे काम करता है

* जब कोई फाइल **चेकआउट (पढ़ना)** की जाती है, तो Git लाइन एंडिंग्स को आपके OS के कन्वेंशन के अनुसार बदल सकता है (ताकि वे आपके एडिटर में सामान्य दिखें)।
* जब कोई फाइल **कमिट (लिखना)** की जाती है, तो Git लाइन एंडिंग्स को वापस LF में बदल सकता है (रिपॉजिटरी में क्रॉस-प्लेटफॉर्म मानक)।

---

### सेटिंग्स

आप इसे इस तरह सेट करते हैं:

```bash
git config --global core.autocrlf <value>
```

सामान्य मान:

* `true`
  चेकआउट पर **LF → CRLF** में बदलें, और कमिट पर **CRLF → LF** में बदलें।
  **Windows डेवलपर्स** के लिए सबसे उपयुक्त जो Windows एडिटर्स में कोड एडिट करते हैं।

* `input`
  चेकआउट पर फाइलों को संशोधित **न करें**, लेकिन कमिट पर **CRLF → LF** में बदलें।
  **Linux/macOS डेवलपर्स** के लिए सबसे उपयुक्त ताकि यह सुनिश्चित हो सके कि रिपॉजिटरी में केवल LF हों।

* `false`
  कोई भी रूपांतरण नहीं। Git फाइलों को ठीक वैसे ही स्टोर और चेकआउट करता है जैसे वे हैं।
  सबसे उपयुक्त अगर आपकी टीम/एडिटर लाइन एंडिंग्स को लगातार हैंडल करती है।

---

### सामान्य सिफारिशें

* **Windows**:

  ```bash
  git config --global core.autocrlf true
  ```
* **Linux/macOS**:

  ```bash
  git config --global core.autocrlf input
  ```

मिश्रित टीमों के लिए, रिपॉजिटरी में एक `.gitattributes` फाइल जोड़ना भी अच्छा होता है ताकि लाइन एंडिंग्स में एकरूपता बनी रहे, उदाहरण के लिए:

```
* text=auto
```

---

क्या आप चाहते हैं कि मैं आपको एक अनुशंसित **`.gitattributes` सेटअप** भी दिखाऊं जो क्रॉस-प्लेटफॉर्म टीमों के लिए अच्छा काम करता है (ताकि आप केवल `core.autocrlf` पर निर्भर न रहें)?
