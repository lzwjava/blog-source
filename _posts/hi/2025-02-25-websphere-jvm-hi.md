WebSphere Liberty Profile (WLP) सर्वर में `jvm.options` फ़ाइल में सेट किए गए JVM विकल्पों को चेक करने के लिए, Java Development Kit (JDK) द्वारा प्रदान की गई `jinfo` कमांड का उपयोग किया जा सकता है। यह तरीका एक चल रहे सर्वर पर लागू किए गए विकल्पों को सत्यापित करने के लिए बहुत प्रभावी है। नीचे दिए गए कदमों का पालन करके यह काम पूरा किया जा सकता है:

### JVM विकल्पों को चेक करने के कदम

1. **संदर्भ समझें**
   सर्वर डायरेक्टरी में `jvm.options` फ़ाइल (आम तौर पर `<WLP_HOME>/usr/servers/<serverName>/jvm.options`) में JVM arguments को स्पेसिफाई किया जाता है, जैसे कि हिप साइज (जैसे कि `-Xmx`), गार्बेज कलेक्शन सेटिंग्स (जैसे कि `-XX:+UseG1GC`), या सिस्टम प्रॉपर्टीज (जैसे कि `-Dmy.property=value`), जो Liberty सर्वर शुरू होने पर लागू होते हैं।

2. **सर्वर शुरू करें**
   सर्वर को पृष्ठभूमि में शुरू करने के लिए निम्न कमांड का उपयोग करें:
   ```
   <WLP_HOME>/bin/server start <serverName>
   ```
   `<WLP_HOME>` को WebSphere Liberty इंस्टॉलेशन का पथ और `<serverName>` को आपका सर्वर का नाम से बदलें। यह कमांड सर्वर को एक पृष्ठभूमि प्रक्रिया के रूप में लॉन्च करता है।

3. **प्रोसेस आईडी (PID) को लॉकेट करें**
   सर्वर शुरू करने के बाद, चल रहे Java प्रक्रिया का प्रोसेस आईडी (PID) आपको चाहिए। Liberty यह कोन्वीनिएन्स के साथ एक `.pid` फ़ाइल में स्टोर करता है, जो यहाँ है:
   ```
   <WLP_HOME>/usr/servers/<serverName>/workarea/<serverName>.pid
   ```
   इस फ़ाइल को खोलें (जैसे कि Unix-like सिस्टम पर `cat` का उपयोग करके या एक टेक्स्ट एडिटर) और PID को प्राप्त करें, जो सर्वर प्रक्रिया का एक संख्यात्मक मान है।

4. **JVM फ्लैग्स को सत्यापित करें**
   चल रहे सर्वर पर लागू किए गए JVM फ्लैग्स को देखने के लिए `jinfo` कमांड का उपयोग करें। चलाएं:
   ```
   jinfo -flags <pid>
   ```
   `<pid>` को `.pid` फ़ाइल से प्राप्त प्रोसेस आईडी से बदलें। यह कमांड JVM को पास किए गए कमांड-लाइन फ्लैग्स को आउटपुट करता है, जैसे कि `-Xmx1024m` या `-XX:+PrintGCDetails`। आउटपुट में देखें कि आपने `jvm.options` में सेट किए गए फ्लैग्स मौजूद हैं।

5. **सिस्टम प्रॉपर्टीज को सत्यापित करें**
   अगर आपका `jvm.options` फ़ाइल में सिस्टम प्रॉपर्टीज शामिल हैं (जैसे कि `-Dmy.property=value`), तो उन्हें अलग से चेक करें:
   ```
   jinfo -sysprops <pid>
   ```
   यह सभी सिस्टम प्रॉपर्टीज को दिखाता है जो JVM के लिए सेट हैं। आउटपुट में आपने परिभाषित प्रॉपर्टीज को खोजें ताकि यह सही तरीके से लागू हो।

### प्रे-रिक्वाइरेमेंट्स
- **JDK इंस्टॉल्ड**: `jinfo` कमांड JDK का हिस्सा है, नहीं JRE का। सुनिश्चित करें कि आपके पास एक JDK इंस्टॉल्ड है और `jinfo` एक्सिक्यूटेबल आपके सिस्टम के PATH में है।
- **अनुमतियाँ**: `jinfo` को उसी यूजर के साथ चलाएं जो सर्वर शुरू किया या प्रक्रिया से जोड़ने के लिए पर्याप्त अधिकार रखता है।

### विकल्प तरीके
अगर `jinfo` उपलब्ध नहीं है या आप किसी और तरीके का पसंद करते हैं, तो यहां दो अतिरिक्त विकल्प हैं:

- **`server dump` का उपयोग करें**:
  1. सर्वर को `server start <serverName>` के साथ शुरू करें।
  2. एक सर्वर डंप जनरेट करें:
     ```
     <WLP_HOME>/bin/server dump <serverName>
     ```
  3. जनरेट किए गए `.zip` फ़ाइल को `<WLP_HOME>/usr/servers/<serverName>/` में लॉकेट करें, इसे एक्सट्रैक्ट करें और `javacore.<timestamp>.txt` फ़ाइल को खोलें।
  4. "CMDLINE" सेक्शन को देखें ताकि पूरा Java कमांड लाइन, जिसमें JVM फ्लैग्स शामिल हैं, और "System Properties" सेक्शन को सिस्टम प्रॉपर्टीज के लिए देखें।

- **स्टार्टअप स्क्रिप्ट को ट्रेस करें** (Unix-like सिस्टम):
  1. ट्रेसिंग के साथ सर्वर स्क्रिप्ट चलाएं:
     ```
     sh -x <WLP_HOME>/bin/server start <serverName>
     ```
  2. आउटपुट को रिव्यू करें ताकि स्क्रिप्ट द्वारा चलाए गए Java कमांड को देखें, जिसमें `jvm.options` से सभी विकल्प शामिल हैं।

### नोट्स
- **सर्वर नहीं चल रहा**: अगर सर्वर गलत विकल्पों के कारण शुरू नहीं होता, तो `server run <serverName>` का उपयोग करके इसे फोरग्राउंड में चलाएं और कंसोल आउटपुट में त्रुटियों को चेक करें, या स्क्रिप्ट ट्रेसिंग तरीके का उपयोग करें ताकि कमांड को देखें।
- **लॉग फ़ाइलें**: `<WLP_HOME>/usr/servers/<serverName>/logs/` में `console.log` फ़ाइल में कुछ स्टार्टअप जानकारी हो सकती है, लेकिन यह सभी JVM विकल्पों को सत्यापित नहीं करती, जब तक कि वर्बोज़ लॉगिंग सक्षम नहीं है (जैसे कि `-XX:+PrintFlagsFinal`).

### निष्कर्ष
`jinfo` तरीका `jvm.options` में सेट किए गए JVM विकल्पों को सत्यापित करने का सबसे सरल तरीका है। ऊपर दिए गए कदमों का पालन करके, आप यह सुनिश्चित कर सकते हैं कि JVM फ्लैग्स और सिस्टम प्रॉपर्टीज सही तरीके से लागू किए गए हैं।

अलावा, `javacore.<timestamp>.txt` फ़ाइल को डंप कमांड का उपयोग करने के बाद भी नहीं मिला।

### WebSphere Liberty 24.0.0.9 में JVM विकल्पों को चेक करने में समस्याओं को सुलझाने

आप WebSphere Liberty Profile (WLP) सर्वर संस्करण 24.0.0.9 में `jinfo` कमांड का उपयोग करके JVM विकल्पों को चेक करने में समस्याओं का सामना कर रहे हैं, जिसमें एक त्रुटि है जो "प्रोसेस से जोड़ने में असमर्थ" कहती है। अलावा, सर्वर डंप जनरेट करने के बाद, आपने अपेक्षित स्थान (`<WLP_HOME>/usr/servers/<serverName>/`) में `javacore.<timestamp>.txt` फ़ाइल को भी नहीं पाया। नीचे दिए गए कदमों का पालन करके, आप इन समस्याओं को सुलझा सकते हैं और सुनिश्चित कर सकते हैं कि आप `jvm.options` फ़ाइल में सेट किए गए JVM विकल्पों को सत्यापित कर सकते हैं।

---

### समस्या समझें

- **`jinfo` विफलता**: "प्रोसेस से जोड़ने में असमर्थ" त्रुटि आम तौर पर `jinfo` को चल रहे Java Virtual Machine (JVM) से जोड़ने में असमर्थ होने का संकेत देती है। यह अक्सर अनुमतियों के मुद्दों (जैसे कि कमांड को उसी यूजर के साथ चलाने के बजाय जो सर्वर शुरू किया) या JVM प्रक्रिया पर प्रतिबंधों के कारण होता है।
- **मिसिंग `javacore.<timestamp>.txt`**: सर्वर डंप प्रक्रिया एक `.zip` फ़ाइल जनरेट करनी चाहिए जिसमें डायग्नोस्टिक फ़ाइलें शामिल हैं, जिसमें `javacore.<timestamp>.txt` भी शामिल है, लेकिन इसका अभाव यह संकेत देता है कि या तो डंप विफल हुआ, या फ़ाइल अपेक्षित स्थान पर नहीं है, या अपेक्षित सामग्री जनरेट नहीं हुई।

क्योंकि आप WebSphere Liberty 24.0.0.9 का उपयोग कर रहे हैं जो एक Unix-like सिस्टम पर लगता है (आम तौर पर फ़ाइल पथों के आधार पर), हम समाधान को इस तरह से तैयार करेंगे।

### कदम-दर-कदम समाधान

JVM विकल्पों को प्राप्त करने के लिए, सबसे सरल विकल्पों से शुरू करके `jinfo` और सर्वर डंप समस्या को सुलझाने के लिए, नीचे दिए गए विकल्प हैं।

#### 1. सर्वर चल रहा है, यह सुनिश्चित करें
पहले, सुनिश्चित करें कि आपका Liberty सर्वर सक्रिय है:

- **कमांड**:
  ```bash
  <WLP_HOME>/bin/server status <serverName>
  ```
- **अपेक्षित आउटपुट**:
  अगर चल रहा है, तो आप एक संदेश देखेंगे जैसे कि "Server <serverName> is running with process ID <pid>." अगर नहीं, तो इसे शुरू करें:
  ```bash
  <WLP_HOME>/bin/server start <serverName>
  ```

- **PID को लॉकेट करें**:
  PID को `<WLP_HOME>/usr/servers/<serverName>/workarea/<serverName>.pid` में लॉकेट करें:
  ```bash
  cat <WLP_HOME>/usr/servers/<serverName>/workarea/<serverName>.pid
  ```
  इस PID को आगे के कदमों के लिए नोट करें।

#### 2. `jps -v` का उपयोग `jinfo` के विकल्प के रूप में
`jps` कमांड (JDK का हिस्सा) Java प्रक्रियाओं और उनके विकल्पों की सूची बनाती है, अक्सर `jinfo` द्वारा सामना किए गए जोड़ने के मुद्दों को पार कर जाती है।

- **कमांड**:
  ```bash
  jps -v
  ```
- **आउटपुट**:
  Java प्रक्रियाओं की सूची, जैसे कि:
  ```
  12345 Liberty -Xmx1024m -XX:+UseG1GC -Dmy.property=value
  ```
- **कार्रवाई**:
  `.pid` फ़ाइल से मिली PID या कमांड लाइन में "Liberty" या आपका `<serverName>` को खोजकर अपने Liberty सर्वर प्रक्रिया को पहचानें। सूचीबद्ध विकल्प (जैसे कि `-Xmx1024m`, `-Dmy.property=value`) में `jvm.options` से विकल्प शामिल हैं।

- **अनुमतियों की जांच**:
  अगर `jps -v` विफल हो जाता है या कोई आउटपुट नहीं दिखाता, तो इसे उसी यूजर के साथ चलाएं जो सर्वर शुरू किया (जैसे कि `sudo -u <serverUser> jps -v`) या `sudo` के साथ:
  ```bash
  sudo jps -v
  ```

#### 3. `jcmd` का उपयोग JVM के लिए विस्तृत जानकारी के लिए
अगर `jps -v` पर्याप्त नहीं है, तो `jcmd` (JDK का एक और उपकरण) एक चल रहे JVM को बिना कुछ `jinfo` के जोड़ने की सीमाओं के पूछताछ कर सकता है।

- **कमांड**:
  - JVM विकल्पों के लिए:
    ```bash
    jcmd <pid> VM.command_line
    ```
    आउटपुट: पूरा कमांड लाइन, जैसे कि `java -Xmx1024m -XX:+UseG1GC -Dmy.property=value ...`
  - सिस्टम प्रॉपर्टीज के लिए:
    ```bash
    jcmd <pid> VM.system_properties
    ```
    आउटपुट: प्रॉपर्टीज की सूची, जैसे कि `my.property=value`.

- **कार्रवाई**:
  `<pid>` को आपका सर्वर का PID से बदलें। सुनिश्चित करें कि आप इन कमांडों को पर्याप्त अधिकार के साथ चलाते हैं (जैसे कि `sudo jcmd <pid> ...` अगर आवश्यक हो).

#### 4. एक जावकोर फ़ाइल जनरेट करें और इसे जांचें
क्योंकि सर्वर डंप `javacore.<timestamp>.txt` फ़ाइल को जनरेट नहीं कर रहा, तो एक स्टैंडअलोन जावकोर फ़ाइल जनरेट करने की कोशिश करें:

- **कमांड**:
  ```bash
  <WLP_HOME>/bin/server javadump <serverName>
  ```
- **अपेक्षित आउटपुट**:
  जावकोर फ़ाइल के स्थान का संदेश, आम तौर पर `<WLP_HOME>/usr/servers/<serverName>/javacore.<timestamp>.txt`.

- **कार्रवाई**:
  - डायरेक्टरी की जांच करें:
    ```bash
    ls <WLP_HOME>/usr/servers/<serverName>/javacore.*.txt
    ```
  - फ़ाइल को खोलें और खोजें:
    - **CMDLINE**: JVM विकल्पों की सूची (जैसे कि `-Xmx1024m`).
    - **System Properties**: `-D` प्रॉपर्टीज की सूची।

- **समस्या सुलझाने**:
  अगर कोई फ़ाइल दिखाई नहीं देती, तो कमांड के दौरान कमांड के कंसोल आउटपुट में त्रुटियों की जांच करें।
  - सुनिश्चित करें कि डंप के दौरान सर्वर चल रहा था (हालांकि `server dump` एक रोके हुए सर्वर पर काम कर सकता है, लेकिन जावकोर एक चल रहे JVM की आवश्यकता होती है).
  - अगर `.zip` फ़ाइल गायब है, तो `<WLP_HOME>/usr/servers/<serverName>/logs/` में सर्वर के लॉग्स में क्लू खोजें।

#### 5. सर्वर डंप तरीके को फिर से देखें
सुनिश्चित करें कि पूरा सर्वर डंप सही तरीके से काम कर रहा है:

- **कमांड**:
  ```bash
  <WLP_HOME>/bin/server dump <serverName>
  ```
- **अपेक्षित आउटपुट**:
  एक `.zip` फ़ाइल जैसे कि `<serverName>.dump-<timestamp>.zip` `<WLP_HOME>/usr/servers/<serverName>/` में।

- **कार्रवाई**:
  - फ़ाइल मौजूद है, यह सुनिश्चित करें:
    ```bash
    ls <WLP_HOME>/usr/servers/<serverName>/*.zip
    ```
  - इसे एक्सट्रैक्ट करें:
    ```bash
    unzip <serverName>.dump-<timestamp>.zip -d temp_dir
    ```
  - `javacore.<timestamp>.txt` को खोजें:
    ```bash
    find temp_dir -name "javacore.*.txt"
    ```
  - फ़ाइल को खोलें और "CMDLINE" और "System Properties" सेक्शन को देखें।

- **समस्या सुलझाने**:
  - कमांड के कंसोल आउटपुट में त्रुटियों की जांच करें।
  - सुनिश्चित करें कि डंप के दौरान सर्वर चल रहा था (हालांकि `server dump` एक रोके हुए सर्वर पर काम कर सकता है, लेकिन जावकोर एक चल रहे JVM की आवश्यकता होती है).
  - अगर `.zip` फ़ाइल गायब है, तो `<WLP_HOME>/usr/servers/<serverName>/logs/` में लॉग्स में क्लू खोजें।

#### 6. वर्बोज़ JVM आउटपुट सक्षम करें (अंतिम विकल्प)
अगर सब कुछ विफल हो जाता है, तो `jvm.options` को सारी JVM फ्लैग्स को लॉग करने के लिए संपादित करें:

- **`<WLP_HOME>/usr/servers/<serverName>/jvm.options` संपादित करें**:
  जोड़ें:
  ```
  -XX:+PrintFlagsFinal
  ```
- **सर्वर को फिर से शुरू करें**:
  ```bash
  <WLP_HOME>/bin/server stop <serverName>
  <WLP_HOME>/bin/server start <serverName>
  ```
- **लॉग्स की जांच करें**:
  `<WLP_HOME>/usr/servers/<serverName>/logs/console.log` खोलें और एक JVM फ्लैग्स की तालिका खोजें, जैसे कि:
  ```
  [uintx] MaxHeapSize = 1073741824 {product}
  ```

### अतिरिक्त नोट्स

- **अनुमतियाँ**:
  `jinfo` विफलता अनुमतियों के मिसमैच का संकेत देती है। सभी कमांडों को उसी यूजर के साथ चलाएं जो सर्वर शुरू किया (जैसे कि `sudo -u <serverUser>` या उस यूजर के रूप में लॉग इन करके) या `sudo` के साथ।
- **WebSphere Liberty 24.0.0.9**:
  डंप व्यवहार डॉक्यूमेंटेड प्रक्रिया के साथ मिलना चाहिए (`.zip` में `javacore.<timestamp>.txt` `<WLP_HOME>/usr/servers/<serverName>/` में)। अगर यह अलग है, तो IBM के डॉक्यूमेंटेशन में संस्करण-विशिष्ट परिवर्तन की जांच करें, हालाँकि `server javadump` तरीका विश्वसनीय बना रहता है।
- **JDK आवश्यकता**:
  सुनिश्चित करें कि आपके पास JDK (नहीं JRE) इंस्टॉल्ड है, क्योंकि `jps`, `jcmd`, और `jinfo` JDK उपकरण हैं।

### सिफारिश किया गया तरीका

1. **पहले `jps -v` का प्रयास करें**: यह तेज है और संभवतः जोड़ने के मुद्दों को टाल सकता है।
2. **`jcmd` पर वापस आ जाएं**: अगर `jps` पर्याप्त नहीं है, तो विस्तृत आउटपुट प्रदान करता है।
3. **`server javadump` का उपयोग करें**: एक जावकोर को सीधे जनरेट करता है, पूरा डंप जटिलता को पार करता है।
4. **डंप को डिबग करें**: अगर आवश्यक हो, फिर से `server dump` चलाएं और `.zip` फ़ाइल के सामग्री की जांच करें।

इन कदमों का पालन करके, आप सफलतापूर्वक अपने JVM विकल्पों को प्राप्त कर सकते हैं और `jinfo` और गायब जावकोर फ़ाइल के साथ समस्याओं को सुलझा सकते हैं। अगर आप किसी विशेष त्रुटि से सामना करते हैं, तो मुझे बता दें!
