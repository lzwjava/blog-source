## 1. फ़ाइल संगठन

### 1.1 फ़ाइल नाम
- स्रोत फ़ाइलों में `.java` एक्सटेंशन होना चाहिए
- स्रोत फ़ाइलनाम फ़ाइल में मौजूद public class के नाम से बिल्कुल मेल खाना चाहिए
- उदाहरण:
  - `MyClass.java` में MyClass नामक एक public class होती है
  - `CircleShape.java` में CircleShape नामक एक public class होती है

### 1.2 फ़ाइल संरचना
1. शुरुआती टिप्पणियाँ
2. पैकेज स्टेटमेंट
3. इम्पोर्ट स्टेटमेंट
4. क्लास और इंटरफेस डिक्लेरेशन

## 2. फॉर्मेटिंग और इंडेंटेशन

### 2.1 इंडेंटेशन
- इंडेंटेशन के लिए 4 स्पेस का उपयोग करें (टैब नहीं)
- कंटिन्यूएशन लाइनों को 8 स्पेस से इंडेंट करें
- ब्रेसिज़ लंबवत रूप से संरेखित होनी चाहिए

### 2.2 लाइन लंबाई
- अनुशंसित अधिकतम लाइन लंबाई 80 वर्ण है
- लंबे स्टेटमेंट के लिए, लाइनों को उच्चतर तार्किक स्तर पर तोड़ें

### 2.3 लाइन रैपिंग
- ऑपरेटर से पहले लाइन तोड़ें
- नई लाइन को समान स्तर पर एक्सप्रेशन की शुरुआत के साथ संरेखित करें

## 3. टिप्पणियाँ

### 3.1 फ़ाइल टिप्पणियाँ
- प्रत्येक स्रोत फ़ाइल एक टिप्पणी ब्लॉक से शुरू होनी चाहिए:
  ```java
  /*
   * Classname
   * 
   * Version information
   * 
   * Date
   * 
   * Copyright notice
   */
  ```

### 3.2 इम्प्लीमेंटेशन टिप्पणियाँ
- मल्टी-लाइन टिप्पणियों के लिए `/* */` का उपयोग करें
- सिंगल-लाइन टिप्पणियों के लिए `//` का उपयोग करें
- टिप्पणियों को "क्या" नहीं, बल्कि "क्यों" समझाना चाहिए

### 3.3 डॉक्यूमेंटेशन टिप्पणियाँ
- क्लास, इंटरफेस, मेथड के लिए Javadoc-स्टाइल टिप्पणियों का उपयोग करें
- शामिल करें:
  - संक्षिप्त विवरण
  - मेथड पैरामीटर के लिए `@param`
  - रिटर्न वैल्यू के लिए `@return`
  - एक्सेप्शन के लिए `@throws`

## 4. डिक्लेरेशन

### 4.1 प्रति लाइन संख्या
- प्रति लाइन एक डिक्लेरेशन
- अनुशंसित: 
  ```java
  int level;        // सही
  int size;         // सही
  
  // Avoid:
  int level, size;  // अनुशंसित नहीं
  ```

### 4.2 इनिशियलाइज़ेशन
- जहाँ संभव हो, डिक्लेरेशन पर वेरिएबल को इनिशियलाइज़ करें
- संबंधित डिक्लेरेशन को एक साथ समूहित करें

## 5. स्टेटमेंट

### 5.1 सरल स्टेटमेंट
- प्रति लाइन एक स्टेटमेंट
- कॉमा के बाद एक स्पेस का उपयोग करें
- ऑपरेटर के आसपास स्पेस का उपयोग करें

### 5.2 कंपाउंड स्टेटमेंट
- ब्रेसिज़ का उपयोग `if`, `else`, `for`, `while`, `do` स्टेटमेंट के साथ किया जाता है, भले ही वह सिंगल-लाइन ब्लॉक हों

### 5.3 रिटर्न स्टेटमेंट
- एक्सप्लिसिट रिटर्न को प्राथमिकता दें
- `return` के बाद अनावश्यक `else` से बचें

## 6. नामकरण सम्मेलन

### 6.1 पैकेज नाम
- सभी लोअरकेस
- यूनिक प्रीफिक्स, अक्सर कंपनी डोमेन उल्टा
- उदाहरण: `com.company.project`

### 6.2 क्लास नाम
- अपरकैमलकेस
- संज्ञा या संज्ञा वाक्यांश
- उदाहरण: `DataProcessor`, `HttpClient`

### 6.3 इंटरफेस नाम
- क्लास नाम के समान
- व्यवहार का वर्णन करने पर विचार करें
- उदाहरण: `Runnable`, `Serializable`

### 6.4 मेथड नाम
- लोअरकैमलकेस
- क्रिया या क्रिया वाक्यांश
- उदाहरण: `calculateTotal()`, `sendMessage()`

### 6.5 वेरिएबल नाम
- लोअरकैमलकेस
- अर्थपूर्ण और वर्णनात्मक
- अस्थायी वेरिएबल को छोड़कर सिंगल-लेटर नामों से बचें

### 6.6 कॉन्स्टेंट नाम
- अंडरस्कोर के साथ ऑल अपरकेस
- उदाहरण: `MAX_VALUE`, `COLOR_RED`

## 7. प्रोग्रामिंग प्रथाएँ

### 7.1 विजिबिलिटी
- संभव हो तो सबसे कम विजिबिलिटी का उपयोग करें
- `public` पर `private` को प्राथमिकता दें
- क्लास फ़ील्ड के लिए एक्सेसर मेथड का उपयोग करें

### 7.2 वेरिएबल स्कोप
- वेरिएबल को संभव हो तो सबसे छोटे स्कोप में डिक्लेयर करें
- वेरिएबल को उनके पहले उपयोग के करीब इनिशियलाइज़ करें

### 7.3 एक्सेप्शन हैंडलिंग
- एक्सेप्शन में हमेशा एक मैसेज शामिल करें
- विशिष्ट एक्सेप्शन को कैच करें
- try-catch-finally का उपयोग उचित रूप से करें

## 8. सर्वोत्तम प्रथाएँ

### 8.1 मैजिक नंबर से बचें
- लिटरल नंबर के बजाय नामित कॉन्स्टेंट का उपयोग करें
- कोड की पठनीयता और रखरखाव में सुधार करें

### 8.2 इम्यूटेबिलिटी को प्राथमिकता दें
- जो वेरिएबल नहीं बदलते हैं, उनके लिए `final` का उपयोग करें
- जहाँ संभव हो इम्यूटेबल क्लास बनाएँ
