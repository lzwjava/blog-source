### Azure वर्चुअल नेटवर्किंग की मूल बातें

Azure अपनी **वर्चुअल नेटवर्क (VNet)** सेवा के माध्यम से एक स्केलेबल और सुरक्षित नेटवर्किंग वातावरण प्रदान करता है, जो AWS VPC के बराबर मूल सेवा है। यह आपके संसाधनों जैसे VMs, App Services, या Kubernetes क्लस्टर्स के लिए Azure क्लाउड में एक निजी, अलग-थलग स्थान बनाता है। नीचे, मैं AWS के साथ समानताओं और अंतरों के साथ प्रमुख घटकों (वर्चुअल नेटवर्क, गेटवे, सबनेट्स, आदि) की व्याख्या आसान तुलना के लिए समान संरचना में करूंगा।

#### वर्चुअल नेटवर्क (VNet)
- **यह क्या है**: Azure में मूलभूत अलग-थलग नेटवर्क, जो एक क्षेत्र में फैला होता है और संसाधनों को निजी तौर पर संचार करने की अनुमति देता है। आप CIDR ब्लॉक्स (जैसे, 10.0.0.0/16) के साथ इसके एड्रेस स्पेस को परिभाषित करते हैं।
- **प्रमुख विशेषताएं**:
  - संसाधनों के लिए डिफ़ॉल्ट आउटबाउंड इंटरनेट एक्सेस (AWS के विपरीत, जहाँ यह ऑप्ट-इन है)।
  - IPv4 और IPv6 का समर्थन करता है; डिफ़ॉल्ट रूप से अलग-थलग।
  - VNet के लिए कोई शुल्क नहीं—गेटवे जैसे संलग्न संसाधनों के लिए भुगतान करें।
- **AWS VPC के समान**: दोनों ही संसाधन अलगाव, स्केलिंग और कनेक्टिविटी के लिए प्राइवेट क्लाउड हैं। Azure VNets स्वचालित रूप से Availability Zones (AZs) में फैलते हैं; AWS VPCs को स्पष्ट AZ सेटअप की आवश्यकता होती है।
- **इसका उपयोग क्यों करें?**: सुरक्षित अंतर-संसाधन संचार, इंटरनेट एक्सेस और ऑन-प्रिमाइसेस लिंक। प्रत्येक Azure सब्सक्रिप्शन को एक्सेस मिलता है, लेकिन नियंत्रण के लिए आप कस्टम VNets बनाते हैं।
- **उदाहरण**: AWS VPC की तरह, यह क्लाउड में आपकी "निजी एस्टेट" है—आप सीमाएं निर्धारित करते हैं, लेकिन Azure आउटबाउंड इंटरनेट जैसे कुछ डिफ़ॉल्ट्स को संभालता है।

#### सबनेट्स
- **ये क्या हैं**: एक VNet के एड्रेस स्पेस के विभाजन, जहाँ संसाधन तैनात किए जाते हैं। प्रत्येक सबनेट VNet के दायरे में होता है और एक क्षेत्र में सभी AZs में फैल सकता है।
- **प्रकार**:
  - **पब्लिक सबनेट**: संसाधनों में इनबाउंड/आउटबाउंड इंटरनेट के लिए पब्लिक IPs हो सकते हैं (Azure Load Balancer या पब्लिक एंडपॉइंट्स के माध्यम से)।
  - **प्राइवेट सबनेट**: कोई सीधा पब्लिक एक्सेस नहीं; डेटाबेस या आंतरिक ऐप्स के लिए आदर्श।
- **प्रमुख विशेषताएं**:
  - CIDR-परिभाषित (जैसे, 10.0.1.0/24)।
  - विभाजन के लिए प्रति VNet कई सबनेट; उनके बीच ट्रैफ़िक को फ़िल्टर किया जा सकता है।
- **AWS सबनेट्स के समान**: दोनों सुरक्षा और संगठन के लिए नेटवर्क को विभाजित करते हैं। Azure का ऑटो-स्पैन AZs HA को सरल बनाता है; AWS सबनेट्स को विशिष्ट AZs से बांधता है।
- **इनका उपयोग क्यों करें?**: वर्कलोड को अलग करता है—जैसे, फ्रंटएंड पब्लिक सबनेट्स में, बैकएंड प्राइवेट में।
- **उदाहरण**: सबनेट्स आपके VNet शहर में "जिले" हैं: सड़क पहुंच (इंटरनेट) वाले सार्वजनिक, दीवारों के पीछे वाले निजी।

#### गेटवे
Azure में गेटवे बाहरी कनेक्टिविटी को संभालते हैं, लेकिन AWS से भिन्न कुछ डिफ़ॉल्ट्स के साथ।

- **इंटरनेट गेटवे समकक्ष**:
  - **यह क्या है**: कोई सीधा IGW नहीं; VNet संसाधनों के लिए आउटबाउंड इंटरनेट डिफ़ॉल्ट रूप से सक्षम है। इनबाउंड के लिए पब्लिक IP या Load Balancer की आवश्यकता होती है।
  - **यह कैसे काम करता है**: ट्रैफ़िक Azure की सिस्टम रूट्स (0.0.0.0/0 से इंटरनेट) के माध्यम से रूट होता है। द्विदिश एक्सेस के लिए पब्लिक IPs का उपयोग करें।
  - **AWS IGW के समान**: दोनों पब्लिक इंटरनेट सक्षम करते हैं, लेकिन Azure आउटबाउंड के लिए अधिक "हमेशा-चालू" है; AWS को स्पष्ट अटैचमेंट और रूट्स की आवश्यकता होती है।
  - **इसका उपयोग क्यों करें?**: वेब ऐप्स के लिए सरल सार्वजनिक एक्सपोजर। बेसिक रूटिंग के लिए मुफ़्त।

- **NAT गेटवे**:
  - **यह क्या है**: एक मैनेज्ड सेवा जो प्राइवेट सबनेट्स से आउटबाउंड-ओनली इंटरनेट (जैसे, VM अपडेट) के लिए एक पब्लिक सबनेट में होती है।
  - **यह कैसे काम करता है**: अनुवाद के लिए एक Elastic IP साझा करता है; AZs में अत्यधिक उपलब्ध।
  - **AWS NAT गेटवे के समान**: दोनों इनबाउंड एक्सपोजर के बिना सुरक्षित आउटबाउंड प्रदान करते हैं। Azure का अधिक एकीकृत और डिफ़ॉल्ट रूप से स्केलेबल है।
  - **इसका उपयोग क्यों करें?**: निजी संसाधनों की सुरक्षा करते हुए एक-तरफ़ा एक्सेस की अनुमति देता है। लागत ~$0.045/घंटा + डेटा।

- **अन्य गेटवे**:
  - **VPN गेटवे**: ऑन-प्रिमाइसेस के लिए साइट-टू-साइट या पॉइंट-टू-साइट VPNs के लिए (AWS VGW की तरह)।
  - **एक्सप्रेसरूट गेटवे**: ऑन-प्रिमाइसेस के लिए निजी, उच्च-बैंडविड्थ लिंक (AWS Direct Connect की तरह)।

#### अन्य संबंधित घटक ("आदि")
- **रूट टेबल्स**: सबनेट ट्रैफ़िक फ्लो को नियंत्रित करते हैं (जैसे, इंटरनेट या पीयर्ड VNets को)। सिस्टम डिफ़ॉल्ट्स मौजूद हैं; कस्टम विशिष्ट रूट्स के लिए ओवरराइड करते हैं। AWS रूट टेबल्स के समान, लेकिन Azure ऑन-प्रिमाइसेस से BGP रूट्स को गतिशील रूप से प्रचारित करता है।
- **नेटवर्क सिक्योरिटी ग्रुप्स (NSGs)**: सबनेट्स या संसाधनों के लिए स्टेटफुल फ़ायरवॉल, जिसमें IP/पोर्ट/प्रोटोकॉल द्वारा नियम होते हैं। AWS सिक्योरिटी ग्रुप्स (इंस्टेंस-लेवल, स्टेटफुल) और NACLs (सबनेट-लेवल, लेकिन Azure के स्टेटफुल हैं) के संयोजन की तरह। लॉजिकल टैगिंग के लिए एप्लिकेशन सिक्योरिटी ग्रुप्स का उपयोग करें।
- **VNet पीयरिंग**: निजी ट्रैफ़िक के लिए VNets (समान/भिन्न क्षेत्र/खाते) को जोड़ता है, AWS VPC पीयरिंग की तरह।
- **सर्विस एंडपॉइंट्स/प्राइवेट लिंक**: इंटरनेट के बिना Azure सेवाओं (जैसे, स्टोरेज) के लिए सुरक्षित निजी एक्सेस, AWS VPC एंडपॉइंट्स के समान।
- **नेटवर्क वर्चुअल एप्लायंसेस**: VNet में VM-आधारित फ़ायरवॉल या राउटर, AWS नेटवर्क फ़ायरवॉल के समान।

#### यह सब एक साथ कैसे काम करता है
1. CIDR के साथ एक VNet बनाएं।
2. सबनेट्स (पब्लिक/प्राइवेट) जोड़ें जो AZs में फैले हों।
3. संसाधनों को डिफ़ॉल्ट आउटबाउंड इंटरनेट मिलता है; इनबाउंड के लिए पब्लिक IPs/Load Balancers जोड़ें।
4. प्राइवेट आउटबाउंड के लिए NAT गेटवे का उपयोग करें।
5. ट्रैफ़िक नियंत्रण के लिए रूट टेबल्स और NSGs लागू करें।
6. हाइब्रिड सेटअप के लिए VNets को पीयर करें या गेटवे जोड़ें।

Azure का मॉडल तेज़ सेटअप के लिए डिफ़ॉल्ट्स (जैसे, आउटबाउंड इंटरनेट) पर जोर देता है, जबकि AWS अधिक विस्तृत ऑप्ट-इन प्रदान करता है। दोनों सुरक्षा और लचीलापन को प्राथमिकता देते हैं।

हाथों-हाथ अनुभव के लिए, Azure पोर्टल के VNet विज़ार्ड को आज़माएं।
