आपका NPM रजिस्ट्री को कूल या इन्सोम्निया जैसी टूल्स के साथ रॉ हट्टप रिक्वेस्ट्स का उपयोग करके टेस्ट करने के लिए, आप इसका कार्यक्षमता को जांचने के लिए कनेक्टिविटी, ऑथेंटिकेशन, और पैकेज डेटा सर्व करने की क्षमता को सत्यापित कर सकते हैं—सभी बिना NPM क्लाइंट पर निर्भर करने के। क्योंकि आपका रजिस्ट्री एक यूजरनेम और पासवर्ड की आवश्यकता करता है, आपको अपने रिक्वेस्ट्स में ऑथेंटिकेशन को मैनुअल रूप से हैंडल करना होगा। नीचे एक पूर्ण गाइड है जो रॉ हट्टप रिक्वेस्ट्स का उपयोग करके अपने रजिस्ट्री को टेस्ट करने के लिए है, मानते हुए कि यह एक NPM-समर्थित रजिस्ट्री है (जैसे कि वेर्डैक्कियो, नेक्सस, या एक कस्टम इम्प्लीमेंटेशन) जो टोकन-आधारित या बेसिक ऑथेंटिकेशन का समर्थन करता है।

---

## रजिस्ट्री को रॉ हट्टप रिक्वेस्ट्स के साथ टेस्ट करने के चरण

### 1. ऑथेंटिकेशन और टोकन प्राप्त करना (अगर आवश्यक हो)

बहुत से NPM रजिस्ट्रियाँ टोकन-आधारित ऑथेंटिकेशन का उपयोग करती हैं, जो आपको आगे के रिक्वेस्ट्स करने से पहले लॉगिन करने के लिए एक टोकन प्राप्त करने की आवश्यकता होती है। कुछ रजिस्ट्रियाँ भी बेसिक ऑथेंटिकेशन का समर्थन कर सकते हैं। यहाँ `कूल` का उपयोग करके ऑथेंटिकेशन करने का तरीका है:

#### `कूल` का उपयोग करके लॉगिन करना
एक टोकन प्राप्त करने के लिए रजिस्ट्री के ऑथेंटिकेशन एंडपॉइंट पर एक PUT रिक्वेस्ट भेजें:

```bash
कूल -X PUT \
  -H "कंटेंट-टाइप: एप्लिकेशन/जेसन" \
  -d '{"नाम": "<यूजरनेम>", "पासवर्ड": "<पासवर्ड>"}' \
  <रजिस्ट्री-यूआरएल>/-/यूजर/ओआरजी.कौचडीबी.यूजर:<यूजरनेम>
```

- **बदलें**:
  - `<यूजरनेम>`: आपका रजिस्ट्री यूजरनेम।
  - `<पासवर्ड>`: आपका रजिस्ट्री पासवर्ड।
  - `<रजिस्ट्री-यूआरएल>`: आपका रजिस्ट्री का पूर्ण यूआरएल (जैसे कि `https://माई-रजिस्ट्री.एक्सैम्पल.कॉम`).
- **अपेक्षित रिस्पॉन्स**: अगर सफल, तो आप एक JSON रिस्पॉन्स प्राप्त करेंगे जिसमें एक टोकन होगा:
  ```json
  {
    "टोकन": "आपका-ऑथ-टोकन"
  }
  ```
- **टोकन को सेभ करें**: `आपका-ऑथ-टोकन` का मान को अगले रिक्वेस्ट्स में उपयोग करने के लिए कॉपी करें।

**नोट**: अगर आपका रजिस्ट्री एक अलग ऑथेंटिकेशन एंडपॉइंट या विधि का उपयोग करता है (जैसे कि बेसिक ऑथ या एक कस्टम एपीआई), तो इसके दस्तावेज़ को चेक करें। अगर यह बेसिक ऑथ को सीधे समर्थन करता है, तो आप इस चरण को छूट सकते हैं और अगले रिक्वेस्ट्स में `-u "<यूजरनेम>:<पासवर्ड>"` शामिल कर सकते हैं।

### 2. रजिस्ट्री को पिंग करना

रजिस्ट्री के साथ बेसिक कनेक्टिविटी को टेस्ट करने के लिए, एक GET रिक्वेस्ट भेजें जिसका लक्ष्य रूट यूआरएल या एक पिंग एंडपॉइंट है।

#### `कूल` का उपयोग करके पिंग करना
```bash
कूल -H "ऑथोराइजेशन: बियरर आपका-ऑथ-टोकन" <रजिस्ट्री-यूआरएल>
```

- **बदलें**:
  - `आपका-ऑथ-टोकन`: चरण 1 से टोकन।
  - `<रजिस्ट्री-यूआरएल>`: आपका रजिस्ट्री यूआरएल।
- **अपेक्षित रिस्पॉन्स**: एक सफल रिस्पॉन्स (HTTP 200) रजिस्ट्री के होमपेज या एक सरल स्टेटस संदेश (जैसे कि `{"डीबी_नाम":"रजिस्ट्री"}` को CouchDB-आधारित रजिस्ट्रियों के लिए) लौट सकता है।
- **अल्टर्नेटिव**: कुछ रजिस्ट्रियाँ एक `/-/पिंग` एंडपॉइंट का समर्थन कर सकते हैं:
  ```bash
  कूल -H "ऑथोराइजेशन: बियरर आपका-ऑथ-टोकन" <रजिस्ट्री-यूआरएल>/-/पिंग
  ```

**बेसिक ऑथ का उपयोग करने के लिए**: अगर आपका रजिस्ट्री टोकन का उपयोग नहीं करता और बेसिक ऑथेंटिकेशन का समर्थन करता है:
```bash
कूल -u "<यूजरनेम>:<पासवर्ड>" <रजिस्ट्री-यूआरएल>
```

### 3. पैकेज मेटाडेटा प्राप्त करना

रजिस्ट्री द्वारा पैकेज मेटाडेटा को सर्व करने की क्षमता को सत्यापित करने के लिए, एक विशिष्ट पैकेज के लिए विवरणों का अनुरोध करें।

#### `कूल` का उपयोग करके मेटाडेटा प्राप्त करना
```bash
कूल -H "ऑथोराइजेशन: बियरर आपका-ऑथ-टोकन" <रजिस्ट्री-यूआरएल>/<पैकेज-नाम>
```

- **बदलें**:
  - `<पैकेज-नाम>`: एक पैकेज जिसका आप जानते हैं कि यह आपका रजिस्ट्री पर मौजूद है (जैसे कि `लॉडैश` अगर यह सार्वजनिक रजिस्ट्री को प्रॉक्सी करता है, या एक निजी पैकेज जैसे कि `माई-ओआरजी-यूटील्स`).
- **अपेक्षित रिस्पॉन्स**: एक JSON ऑब्जेक्ट जिसमें पैकेज के मेटाडेटा, जिसमें संस्करण, निर्भरताएं, और टारबॉल यूआरएल शामिल हैं। उदाहरण के लिए:
  ```json
  {
    "नाम": "लॉडैश",
    "संस्करण": {
      "4.17.21": {
        "डिस्ट": {
          "टारबॉल": "<रजिस्ट्री-यूआरएल>/लॉडैश/-/लॉडैश-4.17.21.तार"
        }
      }
    }
  }
  ```

**बेसिक ऑथ का उपयोग करने के लिए**:
```bash
कूल -u "<यूजरनेम>:<पासवर्ड>" <रजिस्ट्री-यूआरएल>/<पैकेज-नाम>
```

- **सफलता**: एक 200 OK रिस्पॉन्स के साथ मेटाडेटा, यह सत्यापित करता है कि रजिस्ट्री पैकेज डेटा को सही से सर्व कर रहा है।

### 4. एक पैकेज टारबॉल डाउनलोड करना (वैकल्पिक)

रजिस्ट्री को पूरी तरह से टेस्ट करने के लिए, एक पैकेज टारबॉल डाउनलोड करें ताकि यह वास्तविक पैकेज फाइलों को डिलीवर कर सके।

#### `कूल` का उपयोग करके एक टारबॉल डाउनलोड करना
1. चरण 3 में मेटाडेटा से, एक विशिष्ट संस्करण के लिए `टारबॉल` यूआरएल को ढूंढें (जैसे कि `<रजिस्ट्री-यूआरएल>/लॉडैश/-/लॉडैश-4.17.21.तार`).
2. इसे डाउनलोड करें:
```bash
कूल -H "ऑथोराइजेशन: बियरर आपका-ऑथ-टोकन" -O <टारबॉल-यूआरएल>
```

- **बदलें**: `<टारबॉल-यूआरएल>` को मेटाडेटा से यूआरएल से बदलें।
- **`-O` फ्लैग**: फाइल को उसके मूल नाम के साथ सेभ करता है (जैसे कि `लॉडैश-4.17.21.तार`).
- **बेसिक ऑथ का उपयोग करने के लिए**:
  ```bash
  कूल -u "<यूजरनेम>:<पासवर्ड>" -O <टारबॉल-यूआरएल>
  ```
- **सफलता**: फाइल सफलतापूर्वक डाउनलोड होती है, और आप इसे (जैसे कि `तार -xzf <फाइलनाम>` के साथ) निकाल सकते हैं ताकि इसकी सामग्री को सत्यापित कर सकें।

## इन्सोम्निया के साथ टेस्टिंग

अगर आप एक GUI टूल जैसे कि इन्सोम्निया का उपयोग करना पसंद करते हैं, तो इन चरणों का पालन करें:

### 1. ऑथेंटिकेशन सेट अप करना
- इन्सोम्निया में एक नया रिक्वेस्ट बनाएं।
- **ऑथ** टैब पर जाएं:
  - **बियरर टोकन**: अगर आपने चरण 1 में एक टोकन प्राप्त किया है, तो "बियरर टोकन" चुनें और `आपका-ऑथ-टोकन` को पेस्ट करें।
  - **बेसिक ऑथ**: अगर रजिस्ट्री बेसिक ऑथ का उपयोग करता है, तो "बेसिक ऑथ" चुनें और `<यूजरनेम>` और `<पासवर्ड>` दर्ज करें।

### 2. रजिस्ट्री को पिंग करना
- **मेथड**: GET
- **यूआरएल**: `<रजिस्ट्री-यूआरएल>` या `<रजिस्ट्री-यूआरएल>/-/पिंग`
- **सेंड** पर क्लिक करें।
- **अपेक्षित रिस्पॉन्स**: एक 200 OK स्टेटस के साथ एक सरल रिस्पॉन्स बॉडी।

### 3. पैकेज मेटाडेटा प्राप्त करना
- **मेथड**: GET
- **यूआरएल**: `<रजिस्ट्री-यूआरएल>/<पैकेज-नाम>`
- ऑथ टैब में ऑथेंटिकेशन सेट किया जाए, यह सुनिश्चित करें।
- **सेंड** पर क्लिक करें।
- **अपेक्षित रिस्पॉन्स**: एक 200 OK स्टेटस के साथ पैकेज मेटाडेटा JSON में।

### 4. एक टारबॉल डाउनलोड करना
- **मेथड**: GET
- **यूआरएल**: मेटाडेटा से टारबॉल यूआरएल (जैसे कि `<रजिस्ट्री-यूआरएल>/<पैकेज-नाम>/-/<पैकेज-नाम>-<संस्करण>.तार`).
- **सेंड और डाउनलोड** पर क्लिक करें ताकि फाइल को स्थानीय रूप से सेभ करें।
- **सफलता**: फाइल डाउनलोड होती है, यह सत्यापित करती है कि रजिस्ट्री पैकेजों को सर्व कर रहा है।

## समस्या निदान

- **401 अनधिकृत**:
  - आपका टोकन या क्रेडेंशियल्स चेक करें।
  - सुनिश्चित करें कि `ऑथोराइजेशन` हेडर या बेसिक ऑथ सही रूप में फॉर्मेट किया गया है।
- **404 न मिला**:
  - सुनिश्चित करें कि पैकेज रजिस्ट्री पर मौजूद है।
  - अगर सार्वजनिक पैकेजों (जैसे कि `लॉडैश`) को टेस्ट कर रहे हैं, तो सुनिश्चित करें कि रजिस्ट्री सार्वजनिक NPM रजिस्ट्री को प्रॉक्सी करता है।
- **कनेक्टिविटी त्रुटियां**:
  - `<रजिस्ट्री-यूआरएल>` को दोबारा चेक करें (जैसे कि `https://` की आवश्यकता हो तो शामिल करें).
  - HTTPS रजिस्ट्रियों के लिए, अगर `कूल` के साथ SSL त्रुटियां होती हैं, तो `-k` को जोड़ें ताकि सर्टिफिकेट चेक्स को बाईपास करें (केवल टेस्टिंग के लिए).
- **कस्टम रजिस्ट्री व्यवहार**:
  - अगर इन चरणों में विफलता होती है, तो अपने रजिस्ट्री के दस्तावेज़ को चेक करें ताकि विशिष्ट एंडपॉइंट या ऑथेंटिकेशन विधियों के लिए।

## निष्कर्ष

`कूल` या इन्सोम्निया के साथ इन चरणों का पालन करके, आप अपने NPM रजिस्ट्री के कनेक्टिविटी, ऑथेंटिकेशन, और पैकेज-सर्व करने की क्षमता को रॉ हट्टप रिक्वेस्ट्स का उपयोग करके टेस्ट कर सकते हैं। ऑथेंटिकेशन (अगर आवश्यक हो) से शुरू करें, फिर रजिस्ट्री को पिंग करें, एक जाने-माने पैकेज के लिए मेटाडेटा प्राप्त करें, और वैकल्पिक रूप से एक टारबॉल डाउनलोड करें। यह रास्ता सुनिश्चित करता है कि आपका रजिस्ट्री HTTP स्तर पर पूरी तरह से कार्यक्षम है।
