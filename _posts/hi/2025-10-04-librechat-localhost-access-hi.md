### लोकलहोस्ट:3080 पर LibreChat एक्सेस समस्या निवारण

आपके लॉग्स के आधार पर, Docker इमेज सफलतापूर्वक पुल हो गई हैं (कुछ प्रारंभिक टाइमआउट और पुनः प्रयासों के बाद), और 6 कंपोनेंट शुरू हो गए हैं: डिफ़ॉल्ट नेटवर्क, `vectordb`, `chat-mongodb`, `chat-meilisearch`, `rag_api`, और `LibreChat`। हालाँकि, `api` सर्विस (जो आमतौर पर बैकएंड को पोर्ट 3080 पर सर्व करती है) शुरू हुए कंटेनर्स की सूची में नहीं है — यही संभावित कारण है कि आप साइट तक नहीं पहुँच पा रहे हैं। `LibreChat` कंटेनर संभवतः फ्रंटएंड है, जो `api` बैकएंड पर निर्भर करता है।

UID/GID चेतावनियाँ संभावित अनुमति समस्याओं का सुझाव देती हैं जो `api` कंटेनर के ठीक से शुरू होने में रुकावट डाल सकती हैं। यहाँ इसका निदान और समाधान करने के लिए एक चरण-दर-चरण मार्गदर्शिका दी गई है।

#### 1. **चल रहे कंटेनर्स और पोर्ट्स सत्यापित करें**
   सभी सक्रिय कंटेनर्स, उनकी स्थिति और पोर्ट मैपिंग देखने के लिए यह चलाएँ:
   ```
   docker ps
   ```
   - एक `api` कंटेनर ढूंढें (इसका नाम `librechat_api` या इसी तरह का हो सकता है)। यदि यह गायब है या बंद हो गया है, तो यही समस्या है।
   - पुष्टि करें कि क्या पोर्ट `3080` मैप किया गया है (जैसे, `0.0.0.0:3080->3080/tcp`)। यदि नहीं, तो सर्विस इसे एक्सपोज़ नहीं कर रही है।
   - यदि कोई भी कंटेनर पोर्ट 3080 नहीं दिखाता है, तो अगले चरणों के साथ आगे बढ़ें।

#### 2. **कंटेनर लॉग्स जांचें**
   शुरू होने में त्रुटियों के लिए, विशेष रूप से `api` और `LibreChat` सर्विसेज के लिए लॉग्स का निरीक्षण करें:
   ```
   docker logs LibreChat
   docker logs api  # या अगर नाम अलग है तो docker logs librechat_api
   docker logs rag_api  # निर्भरता समस्याओं के मामले में
   ```
   - सामान्य त्रुटियाँ: Permission denied (UID/GID के कारण), MongoDB/Meilisearch कनेक्शन विफलताएँ, या बाइंडिंग समस्याएँ (जैसे, 0.0.0.0 पर सुनना नहीं)।
   - यदि लॉग सर्वर के शुरू होने को दिखाते हैं लेकिन केवल कंटेनर के अंदर localhost पर बाइंड कर रहे हैं, तो अपनी `.env` फ़ाइल में `HOST=0.0.0.0` जोड़ें।

#### 3. **अनुमति चेतावनियों को ठीक करने के लिए UID और GID सेट करें**
   आपकी `.env` फ़ाइल (जो `.env.example` से कॉपी की गई है) में ये वेरिएबल संभवतः कमेंट आउट हैं। अनसेट वेरिएबल फ़ाइल अनुमति बेमेल के कारण कंटेनर्स को चुपचाप विफल कर सकते हैं।
   - `.env` एडिट करें:
     ```
     UID=1000  # अपना यूजर ID पाने के लिए `id -u` चलाएँ
     GID=1000  # अपना ग्रुप ID पाने के लिए `id -g` चलाएँ
     ```
   - सेव करें, फिर रीस्टार्ट करें:
     ```
     docker compose down
     docker compose up -d
     ```
   यह सुनिश्चित करता है कि वॉल्यूम (जैसे config/logs) आपके यूजर के स्वामित्व में हों।

#### 4. **कनेक्टिविटी टेस्ट करें**
   - जांचें कि क्या पोर्ट 3080 स्थानीय रूप से सुन रहा है:
     ```
     curl -v http://localhost:3080
     ```
     - यदि यह टाइमआउट हो जाता है या कनेक्शन से इनकार करता है, तो सर्विस चल/एक्सपोज़ नहीं हो रही है।
   - यदि `docker ps` पोर्ट मैप्ड दिखाता है लेकिन curl विफल हो जाता है, तो फ़ायरवॉल जांचें (जैसे, `sudo ufw status`) या `http://127.0.0.1:3080` आज़माएँ।

#### 5. **आवश्यकता पड़ने पर अतिरिक्त सुधार**
   - **इमेज पुल समस्याएँ**: आपके पहले प्रयास में `ghcr.io/v2/librechat/librechat/manifests/latest` के लिए "denied" त्रुटि थी। यदि पुल फिर से विफल होते हैं, तो GitHub Container Registry के साथ प्रमाणित करें:
     ```
     echo $YOUR_GITHUB_TOKEN | docker login ghcr.io -u YOUR_GITHUB_USERNAME --password-stdin
     ```
     (github.com/settings/tokens पर `read:packages` स्कोप के साथ एक पर्सनल एक्सेस टोकन बनाएँ।)
   - **अनुशंसित Compose फ़ाइल का उपयोग करें**: डिफ़ॉल्ट `docker-compose.yml` लोकल बिल्ड्स के लिए है, जो अनिश्चित हो सकती है। स्थिर रिमोट इमेज के लिए `deploy-compose.yml` पर स्विच करें:
     - अपनी मौजूदा `docker-compose.yml` का बैकअप लें या उसका नाम बदलें।
     - इसके बजाय `deploy-compose.yml` का उपयोग करें: `docker compose -f deploy-compose.yml up -d`.
     - सुनिश्चित करें कि `.env` में `HOST=0.0.0.0` और `NODE_ENV=production` है।
   - **प्रून के साथ पूर्ण रीस्टार्ट**: यदि अटक गए हैं, तो सफाई करें:
     ```
     docker compose down -v  # वॉल्यूम भी हटाता है
     docker system prune -f
     docker compose up -d
     ```
   - **निर्भरताएँ**: पुष्टि करें कि MongoDB और Meilisearch स्वस्थ हैं (`docker logs chat-mongodb`, आदि)। `api` उन पर निर्भर करता है।

इन चरणों के बाद, http://localhost:3080 तक पहुँचने का पुनः प्रयास करें। यदि लॉग विशिष्ट त्रुटियाँ दिखाते हैं (जैसे, Mongo कनेक्शन), तो अधिक लक्षित सहायता के लिए उन्हें साझा करें। एक बार UID/GID और निर्भरताएँ ठीक हो जाने पर यह सेटअप अधिकांश उपयोगकर्ताओं के लिए काम कर जाता है।
