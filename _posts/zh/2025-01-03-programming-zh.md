* 编程竞赛是好的，只要它能激励你。

* 编程就像写作，编程是一种创造性活动。

* 做自己的项目。写你的技术博客。为你将维护多年的项目编程，就像维护一个十年长博客一样。

* 通常，你不需要追逐当前技术中的热门趋势，因为许多趋势几年后会消退。

* 追求你的好奇心，为了自己的缘故编程。

* 尝试为自己创建程序，它们不是工作任务。

* 如果你在编程时经常感到不快乐，那么你做错了。

* iOS、Android、后端、前端、AI 都是好的。至少可以尝试用它们做一个小项目，或者学习它们几个月。

* 调试就是怀疑。不要相信每一行代码，你可以想到更好的方法来做。

* 在编程中，即使是一个字符或一行日志也很重要。它们告诉你一些信息。

* 编程使产品供他人使用。拥有用户是有趣的。

* 不需要严肃。几百个真正喜爱你产品的用户比几万个只是友好地喜欢你产品的用户更好。

* 记住你为什么开始编程，永远不要忘记它。

* 将编程中的知识应用到生活的各个方面。它们是相同的。分批处理事务或一次处理一件。如何将工作分解为单位。每个应用程序背后的基础技术。网络请求背后的细节。

* 抽象和逻辑思维。细致入微的思维。每个解决方案的思考。

* 真理就是真理。通常，计算机不会出错。电路不会出错。编译器不会出错。出现错误时不要感到沮丧。

* 追求优雅和简单的解决方案。简洁是最高的智慧。你需要努力想办法保留本质的东西，去除不必要的东西。

* 对于编程语言，能完成工作的语言就可以了。我个人推荐 Java 和 Python。

* 关注 Yin Wang 的 [https://www.yinwang.org](https://www.yinwang.org)。他是编程界为数不多的天才之一，尽管他自己说天才从不存在。

* 编程的知识和原则可以很容易地应用到语言学习、硬件修复、生活技巧和科学研究。

* 对于大多数编程任务，你不需要高于高中数学的数学。

* 多年后反思你的旧代码，或者长期维护一个代码项目。这会教给你很多东西。

* 如果你失去了编程的热情，就做一些其他事情。

* 测试的时机很重要。自然去做它。你经常不需要为项目编写测试。尽量不要编写测试，编写单元测试，编写集成测试，编写 API 测试。明智地比较它们。

* 尝试 AI 代码编辑器。经常使用 ChatGPT 或其他聊天机器人。由于 AI 工具现在很容易使用，你可以专注于更具有创造性或更重要的部分。

* 调试时，检查是否使用了库的最新版本。如果库不再维护，寻找有活跃维护的克隆或分支。

* 在改善网络速度或程序运行时间时，必须有一个量化指标。否则，你不会知道是否有轻微的改进或退化。

* 对于个人项目，不写测试代码是可以的，但对改变大量代码后的本地测试更好。考虑受影响的代码，它在云管道中的运行时间，以及它可能导致错误的频率，然后相应地编写测试代码。使用可以轻松测试而不会对用户体验产生负面影响的方法。

* 编写简单而优雅的代码。最小化重复，虽然有时重复导致更简单的解决方案。最小化特殊情况。使其易于测试。重构以使用常见的函数或过程，使用递归或循环，并寻找模式。

* 正确处理错误。考虑根本原因，责任，以及我们是否可以改变它，或者它是否是外部错误。考虑救援方法，影响范围，何处处理它，是否应分类错误，它发生的可能性以及最坏的情况。

* 使用 replace 与使用 startWith 后跟 slice 的区别是前者不考虑字符串的位置。将类似的思维应用于编程的每一个细节。

* 最小化一个项目的可能值；对于一个案例仅使用一个值。如果我们已经有了 false，不要使用 null。如果我们有一个翻译的 true 或 false 标志，我们需要确保每次我们有一个翻译标志时，我们不将翻译标志的不存在视为 false。

* 使用 GitHub 或 Sourcetree 经常查看更改的代码块。它们更方便阅读代码。

* 在编程中，通常没有琐碎的事。每个字符、列表项的顺序、每个字符串、每个数字和每个变量名都很重要。每个执行顺序和每个日志都很重要。
