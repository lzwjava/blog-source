## 1. ファイル構成

### 1.1 ファイル名
- ソースファイルは `.java` 拡張子を持つこと
- ソースファイル名はファイル内のpublicクラス名と完全に一致すること
- 例:
  - `MyClass.java` は `MyClass` というpublicクラスを含む
  - `CircleShape.java` は `CircleShape` というpublicクラスを含む

### 1.2 ファイル構造
1. 先頭コメント
2. パッケージ文
3. インポート文
4. クラスとインターフェースの宣言

## 2. フォーマットとインデント

### 2.1 インデント
- インデントには4つのスペースを使用（タブは使用しない）
- 継続行は8つのスペースでインデント
- ブレースは垂直方向に揃えること

### 2.2 行の長さ
- 推奨される最大行長は80文字
- 長い文の場合、より高い論理レベルで行を分割する

### 2.3 行の折り返し
- 演算子の前で行を分割する
- 新しい行を同じレベルの式の先頭に揃える

## 3. コメント

### 3.1 ファイルコメント
- すべてのソースファイルはコメントブロックで始めること:
  ```java
  /*
   * クラス名
   * 
   * バージョン情報
   * 
   * 日付
   * 
   * 著作権表示
   */
  ```

### 3.2 実装コメント
- 複数行コメントには `/* */` を使用
- 単一行コメントには `//` を使用
- コメントは「何を」ではなく「なぜ」を説明すること

### 3.3 ドキュメントコメント
- クラス、インターフェース、メソッドにはJavadocスタイルのコメントを使用
- 以下を含める:
  - 簡単な説明
  - メソッドパラメータの `@param`
  - 戻り値の `@return`
  - 例外の `@throws`

## 4. 宣言

### 4.1 1行あたりの数
- 1行に1つの宣言
- 推奨:
  ```java
  int level;        // 正しい
  int size;         // 正しい
  
  // 避ける:
  int level, size;  // 推奨しない
  ```

### 4.2 初期化
- 可能な場合は宣言時に変数を初期化する
- 関連する宣言をグループ化する

## 5. 文

### 5.1 単純文
- 1行に1つの文
- カンマの後にスペースを使用
- 演算子の周囲にスペースを使用

### 5.2 複合文
- ブレースは `if`、`else`、`for`、`while`、`do` 文で使用し、単一行ブロックでも使用する

### 5.3 return文
- 明示的なreturnを推奨
- `return` 後の不要な `else` を避ける

## 6. 命名規則

### 6.1 パッケージ名
- すべて小文字
- 一意のプレフィックス、多くの場合は会社のドメインを逆順に
- 例: `com.company.project`

### 6.2 クラス名
- アッパーキャメルケース
- 名詞または名詞句
- 例: `DataProcessor`, `HttpClient`

### 6.3 インターフェース名
- クラス名と同様
- 動作を表すことを考慮
- 例: `Runnable`, `Serializable`

### 6.4 メソッド名
- ローワーキャメルケース
- 動詞または動詞句
- 例: `calculateTotal()`, `sendMessage()`

### 6.5 変数名
- ローワーキャメルケース
- 意味があり説明的な名前
- 一時変数を除き、1文字の名前は避ける

### 6.6 定数名
- すべて大文字でアンダースコア区切り
- 例: `MAX_VALUE`, `COLOR_RED`

## 7. プログラミング実践

### 7.1 可視性
- 可能な限り低い可視性を使用
- `public` より `private` を優先
- クラスフィールドにはアクセサメソッドを使用

### 7.2 変数スコープ
- 可能な限り最小のスコープで変数を宣言
- 変数の初期化は最初の使用場所に近い場所で行う

### 7.3 例外処理
- 例外には常にメッセージを含める
- 特定の例外をキャッチする
- try-catch-finallyを適切に使用する

## 8. ベストプラクティス

### 8.1 マジックナンバーの回避
- リテラル数値の代わりに名前付き定数を使用
- コードの可読性と保守性を向上させる

### 8.2 不変性の優先
- 変更されない変数には `final` を使用
- 可能な場合は不変クラスを作成

## 結論
これらの規約は、Javaプロジェクト全体でのコードの可読性、保守性、一貫性を向上させることを目的としています。
