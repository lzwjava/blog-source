### 概要
GitHub Copilotは、VS CodeやVisual StudioなどのIDEでコード支援を行ういくつかのモードを提供しています。これにはEdit Mode（特定の行や関数を選択して修正する「修正モード」に相当）とAgent Mode（ファイルやプロジェクト全体にわたる自律的な広範な編集を処理する）が含まれます。Edit Modeは正確でユーザー制御が強く、対象を絞った修正に最適です。一方、Agent ModeはAIペアプログラマーのように振る舞い、複雑なタスクを推論し、絶え間ない入力を必要とせずにマルチファイルの変更を行います。[1][1] 両モードとも生産性を向上させますが、適用範囲、自律性、ワークフローが異なります。

### 主な違い
Edit Modeはユーザーが選択したコードスニペットに焦点を当て、変更を適用する前にレビューと承認を求める提案を提供します。対照的に、Agent Modeはより高いレベルで動作し、コードベース全体のコンテキストを分析して、編集を自律的に計画、実行、反復し、一貫性を保つためにファイル全体や関連コンポーネントを変更することがよくあります。[2][1] 以下に並べて比較します：

| 機能                     | Edit Mode (Fix Mode)                                                                 | Agent Mode                                                                 |
|--------------------------|--------------------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **適用範囲**             | 選択された行、関数、または単一ファイルに限定。バグ修正、リファクタリング、特定部分の改善のためにコードをハイライトします。[1] | ワークスペースまたはプロジェクト全体。選択範囲を超えて、関連ファイルを自動的に特定し編集します。[2][3] |
| **ユーザー制御**         | 高い：変更を提案し、ユーザーのレビューと明示的な承認を求めます。編集内容を正確に定義します。[4] | 中程度：編集を自動的に適用しますが、リスクの高いコマンド（例：ターミナル実行）はレビューのためにフラグを立てます。目標は自然言語プロンプトで設定します。[1][1] |
| **自律性**               | 低い：対象を絞った提案を提供します。ファイル間の推論や独立したアクションの実行は行いません。[1] | 高い：段階的に推論し、テスト/コマンドを実行し、エラーを検出し、自己修正します。セッション間でコンテキストを維持します。[2][3] |
| **応答時間**             | 速い：選択部分のみを素早く分析します。[2] | 遅い：プロジェクト全体のコンテキストを分析するため、大規模なコードベースでは時間がかかる場合があります。[2] |
| **最適な用途**           | 関数のデバッグ、ループの最適化、メソッドの書き換えなど、広範な影響を与えない迅速な修正。[1] | ファイル間のリファクタリング、モジュールのテスト生成、コード移行、ゼロからの機能構築などの複雑なタスク。[3][5] |
| **例**                   | - バグのある関数を選択：「このnullチェックを修正して」<br>- 行をハイライト：「これを非同期にして」 [2] | - プロンプト：「サービスレイヤー全体をasync/awaitを使用するようにリファクタリングし、すべての依存関係を更新して」<br>- または：「このJavaプロジェクトをファイル全体でJDK 21に近代化して」 [5][6] |
| **リスク/制限事項**      | 変更が分離されているためリスクは最小限。ただし、各修正に対して手動で選択する必要があります。[1] | 自律性が高いため意図しない変更が生じる可能性があります。差分は常にレビューしてください。制御が厳格な環境には不向きです。[7][4] |

### ユースケースとワークフロー
- **対象を絞った修正にはEdit Mode**: 例えば、バグの修正やパフォーマンス向上のために、関数内の問題のあるコードを選択するなど、何が問題か正確にわかっている場合に使用します。これは「スポット編集」ツールのようなものです。IDEでコードを選択し、チャット（例: "@workspace /fix"）でCopilotにプロンプトを送信し、差分プレビューを適用します。このモードは、影響を受けない領域を大幅に変更することなく完全な制御を維持したい反復開発で威力を発揮します。例えば、.NETプロジェクトでメソッドを選択し、「null参照例外を特定して修正案を提案して」とそのスニペットのみに対して依頼する場合などです。[2][8] これはVS CodeおよびVisual StudioのGitHub Copilot拡張機能で利用可能です。

- **プロジェクト全体の編集にはAgent Mode**: ファイル全体の編集やコードベース全体での更新調整が必要な場合など、包括的な変更に対してこのモードをアクティベートします。Copilot Chat（例: "#agentmode" またはドロップダウン経由）でセッションを開始し、「非推奨のAPIの使用をすべて見つけて、このプロジェクト内で新しいものに移行して」のような高レベルのプロンプトを与えると、ステップを計画します：ファイルの分析、編集の提案、テストの実行、反復。新しいファイルの作成、名前空間の更新、アプリのセクションのスキャフォールディングさえも行うことができます。例えば、Javaの近代化では、レガシープロジェクトをスキャンし、Gradleの依存関係を更新し、複数ファイルにわたって変更を検証します。[5][3] このモードは、リファクタリング、大規模なバグ調査、ドキュメントやテストの追加などの反復タスクの自動化に特に強力です。[6][9]

Agent ModeはEdit Modeの基盤の上に構築されていますが、それを拡張しています。Editを精密な切り取りのためのメスと考え、Agentを手術全体を扱う外科医と考えてください。[1] カスタム指示（例：VS Code設定経由）を使用して、命名規則の強制など、一貫性を持たせてAgent Modeを導くことができます。[1]

### それぞれを選択するタイミング
- タスクが局所的（例：単一関数の修正）である場合は、**Edit/Fix Mode**を選択して、シンプルかつ迅速に保ちます。
- ファイル全体の編集や多段階のリファクタリングなど、より大きな範囲で効率を上げるには、**Agent Mode**に切り替えます。ただし、その出力を信頼するために、小さく始めてみてください。[2][1] 両方ともGitHub Copilotのサブスクリプション（Pro以上）が必要で、VS Code/Visual Studioに統合されています。最良の結果を得るには、それらを組み合わせてください：初期修正にEditを使用し、その後、変更をプロジェクト全体に伝播させるためにAgentを使用します。[8] 機密性の高いコードを扱う場合は、Agentの提案すべてを注意深くレビューして、予期しない事態を避けてください。[7]
