### IBM WebSphere Application Server におけるサービス統合バス

IBM WebSphere Application Server (WAS) の文脈における「統合バス」は、通常、WAS によって提供される組み込みのメッセージング基盤である **サービス統合バス (SIB または SIBus)** を指します。これは、メッセージベースのアーキテクチャやサービス指向アーキテクチャ (SOA) を使用するアプリケーション間で、信頼性の高い非同期通信を実現します。

#### 目的
SIB は、WAS 環境内で仮想のメッセージング基盤として機能します。異なるサーバーやクラスター上で動作するアプリケーションが、直接的なポイントツーポイント接続なしでメッセージを交換できるようにし、疎結合、スケーラビリティ、フォールトトレランスを促進します。主なユースケースは以下の通りです：
- キューイングおよびパブリッシュ/サブスクライブパターンのための Java Message Service (JMS) のサポート。
- SOA 設定におけるエンタープライズサービスの統合。
- 分散システム全体でのメッセージルーティング、変換、永続性の処理。

IBM Integration Bus (旧 WebSphere Message Broker) のようなスタンドアロンのエンタープライズサービスバス (ESB) とは異なり、SIB は WAS にネイティブに組み込まれており、別途インストールは不要で、設定を通じて有効化されます。

#### 主要コンポーネントとアーキテクチャ
- **バスメンバー**: バスに参加する WAS セル内のアプリケーションサーバーまたはサーバークラスター。各メンバーはメッセージング基盤の一部をホストします。
- **メッセージングエンジン (ME)**: メッセージを処理するコアランタイムコンポーネント。各 ME は WAS プロセス内 (例えば、バスメンバー上) で動作し、メッセージの送信、受信、保存を処理します。ME は動的に接続され、高可用性を実現する仲介ネットワークを形成します。
- **SIB サービス**: 各 WAS アプリケーションサーバー上のデフォルトサービスで、初期状態では無効になっています。これを有効にするとメッセージング機能が活性化されます。
- **デスティネーション**: メッセージがパブリッシュまたはコンシュームされるキューまたはトピック。WAS 管理コンソールから設定可能です。
- **データストア**: 永続性のために、ME はファイルベースのストア (単一サーバー用はローカル、クラスター用は共有ファイルシステム) またはデータベースを使用し、メッセージの耐久性を確保します。

このアーキテクチャはセルベースです：WAS Network Deployment 設定では、複数のバスメンバーがノード間で連携し、相互運用性のために SOAP/HTTP や JMS などのプロトコルを使用します。

#### 設定方法と使用方法
1.  **作成**: WAS Integrated Solutions Console (管理コンソール) で、*サービス統合 > バス > 新規* に移動します。バス名を定義し、メンバー (サーバー/クラスター) を追加し、ME (高可用性ポリシーなど) を設定します。
2.  **設定**: SIB JMS プロバイダーを有効にし、デスティネーション (キュー/トピック) を作成し、アプリケーションがアクセスできるように JNDI 名にマッピングします。
3.  **使用**: アプリケーションは Message-Driven Bean (MDB) をデプロイするか、JMS API を使用してバスと対話します。例えば、送信側アプリケーションがキューにメッセージを配置すると、バスが他のメンバー上の受信側にルーティングします。

SIB は JMS 1.1/2.0 などの標準をサポートし、外部システム (WebSphere MQ ブリッジ経由など) と統合できます。本番環境では、フェイルオーバーのための共有データストアの使用など、パフォーマンスチューニングを検討してください。

**注意**: *IBM Integration Bus* (IIB) を指している場合、それはより広範な ESB ニーズに対応する別個の製品であり、WAS/SIB に接続できますが、WAS 自体に「含まれる」ものではありません。
