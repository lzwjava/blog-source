## 1. ソースファイルの基本
- **ファイル命名**: 
  - ソースファイルは、含まれるトップレベルクラスの大文字小文字を区別した名前で命名する
  - ファイルはUTF-8エンコーディングを使用すること
  - ファイル名での特殊文字の使用は推奨しない

## 2. ソースファイルの構造
### ソースファイルの構成
1. パッケージ文
2. インポート文
3. トップレベルクラスは1つのみ

### インポート文のルール
- ワイルドカードインポートは使用しない
- 静的インポートは許可される
- インポートは特定の順序で整理する:
  - すべての静的インポート
  - すべての非静的インポート
  - 各グループ内のインポートはアルファベット順

## 3. フォーマットガイドライン

### インデントとブレース
- インデントには2スペースを使用する（タブは不可）
- `if`、`else`、`for`、`do`、`while`文では、1行ブロックでもブレースを使用する
- K&Rスタイルのブレース配置を推奨
  ```java
  public void method() {
    if (condition) {
      // コードブロック
    }
  }
  ```

### 行の長さと折り返し
- 最大行長は100文字
- 行の折り返しはより高位の構文の区切りで行うことが望ましい
- メソッド呼び出しチェインを折り返す場合、`.`の前で折り返す

## 4. 命名規則

### 一般的なルール
- **パッケージ**: 小文字、アンダースコアなし
- **クラス**: アッパーキャメルケース
- **メソッド**: ローワーキャメルケース
- **定数**: アッパースネークケース
- **非定数フィールド**: ローワーキャメルケース
- **パラメータ**: ローワーキャメルケース
- **ローカル変数**: ローワーキャメルケース

### 具体的な命名方法
- 省略語は避ける
- 例外名は`Exception`で終わること
- テストクラスは`TestClassName`と命名する

## 5. プログラミング実践

### Java言語ルール
- **例外**: 
  - 特定の例外をキャッチする
  - 空のcatchブロックを避ける
  - 常に詳細なエラーメッセージを含める
- **finalキーワード**: 
  - メソッドパラメータには`final`を使用する
  - 不変オブジェクトを優先する
- **アノテーション**: 
  - オーバーライドするメソッドには`@Override`が必須
  - 標準アノテーションを適切に使用する

### コード構造
- 継承よりコンポジションを優先する
- メソッドは短く焦点を絞って作成する
- 1行に1つの文
- 条件文の深いネストを避ける

## 6. コメントとドキュメント

### Javadocガイドライン
- すべてのpublicクラスとメソッドにはJavadocが必要
- Javadocコメントは`/**`で始める
- 説明、パラメータ、戻り値、例外を含める
- `@param`、`@return`、`@throws`タグを使用する

### インラインコメント
- 控えめに使用する
- 「何を」ではなく「なぜ」を説明する
- コメントは簡潔で意味のあるものにする

## 7. バージョン管理の考慮事項
- コメントアウトされたコードのコミットは避ける
- 変更の追跡にはコードコメントではなくバージョン管理システムの機能を使用する

## 8. パフォーマンスの考慮事項
- 拡張forループを優先する
- 文字列連結には`StringBuilder`を使用する

- 不必要なオブジェクトの作成を避ける
