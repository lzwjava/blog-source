* 競爭性編程只要能激勵你即可。

* 編程就像寫作。編程是一項創造性活動。

* 自己做項目。寫技術部落格。為一個你會維護多年的項目編程，就像維護一個十年的部落格。

* 通常，你不需要追求現在科技的熱門趨勢，因為很多趨勢幾年後就會消退。

* 追求你的好奇心，為了自己而編程。

* 嘗試為自己創建程序。它們不是工作任務。

* 如果你經常在編程時感到不快樂，那你就做錯了。

* iOS、Android、後端、前端、AI 都很好。可以嘗試使用它們做一個小項目或學習幾個月。

* 除錯是懷疑。不要相信每一行的程式碼，你可以想出更好的方法來做。

* 在編程中，就算是一個字符或一行的日誌記錄也很重要。它們告訴你一些事。

* 使用編程來為他人製作產品。擁有用戶是很有趣的。

* 你不需要對自己太嚴苛。幾百個真正喜愛你產品的用戶比幾萬個只是喜歡你產品的用戶好。

* 應該記得你為什麼會進入編程，永遠不要忘記。

* 將編程中的知識應用到生活的每個方面。它們是一樣的。以批量或逐一的方式做事。如何將工作分為單位。每個應用程式背後的技術基礎。網路請求背後的細微差異。

* 抽象和邏輯思維。細緻入微的思維。每個解決方案的思考。

* 真理就是真理。通常，電腦不會錯。電路不會錯。編譯器不會錯。當有錯誤時，不要感到沮喪。

* 追求優雅且簡單的解決方案。簡單是最高的智慧。你需要仔細思考留下什麼是本質的，去掉什麼是多餘的。

* 至於編程語言，能完成工作的語言就行。我個人推薦 Java 和 Python。

* 關注 Yin Wang: [https://www.yinwang.org](https://www.yinwang.org)。他是編程中少數幾個天才之一，雖然他說天才從來不存在。

* 編程的知識和原則可以輕鬆應用到語言學習、硬體修理、生活技巧和科學研究。

* 對於大多數編程任務，除了高中數學，不需要華而不實的數學。

* 反思幾年後的舊程式碼，或維護一個長期的程式碼項目。這會教你很多。

* 如果你失去了編程的熱情，只需暫時做一些其他事情。

* 測試的時間很重要。自然地進行測試。你經常不需要為項目寫測試。嘗試不寫測試，寫單元測試，寫整合測試，寫API測試。明智地比較它們。

* 嘗試使用AI代碼編輯器。經常使用 ChatGPT 或其他聊天機器人。由於AI工具現今易於使用，你可以專注於更有創造性或更重要的部分。

* 除錯時，檢查是否使用最新版本的庫。如果一個庫未維護，尋找活躍維護的克隆或分支。

* 當改進像網路速度或程序運行時間之類的東西時，必須有一個量化指標。否則，你無法確定是否有微小的改進或惡化。

* 對於個人項目，不寫測試代碼是可以的，但改變大量代碼後進行本地測試是更好的。考慮受影響的代碼、它在雲端管道中運行的時間以及它可能導致錯誤的頻率，然後根據情況編寫測試代碼。使用能夠進行容易測試而不影響用戶體驗的方法。

* 寫簡單和優雅的代碼。最小化重複，雖然有些時候重複會導致更簡單的解決方案。最小化特殊情況。讓它易於測試。重構以使用通用函數或過程，使用遞歸或循環，並尋找模式。

* 正確處理錯誤。思考根本原因，責任，我們是否可以改變它或者它是否是外部錯誤。考慮救援方法，影響範圍，處理它的位置，我們是否應該分類錯誤，它發生的可能性以及最壞的情況。

* 使用 replace 和使用 startWith 後跟 slice 的區別在于前者不考慮字符串的位置。將類似的思維應用到編程的每個細節。

* 最小化一個項目的可能值，針對一個情況使用一個值。如果我們已經有 false，就不要使用 null。如果我們有一個轉譯後的 true 或 false 旗標，我們需要確保每次都有轉譯後的旗標，不要將未轉譯的旗標視為 false。

* 經常使用 GitHub 或 Sourcree 來查看更改的代碼區塊。它們對於閱讀代碼更方便。

* 在編程中，通常沒有琐碎的事。每個字符、列表條目的順序、每個字符串、每個數字和每個變量名都很重要。每個執行順序和每個日誌都很重要。
