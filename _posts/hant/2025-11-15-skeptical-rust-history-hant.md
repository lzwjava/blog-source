#### 初遇契機（2009年，印第安納大學）
- 作者在印第安納大學求學期間，有兩位非電腦科學本科的同學在 Dan Friedman 的高階程式語言課程中表現掙扎。

- 2009年夏季，他們進入 Mozilla 研究院實習，參與 Rust 語言的早期開發（當時應是 Graydon Hoare 的個人專案，後被 Mozilla 採納；推測這兩位同學屬最早期的貢獻者或暑期實習生）。
- 暑期結束後，他們在校內舉辦講座向眾人介紹 Rust，這也是作者首次接觸該語言。

#### 2009年講座（作者的第一印象）
- 講座充滿行銷話術：口號響亮卻缺乏技術實質內容。
- 他們展示了一張列有「Rust三大特性」的三角圖——其中一項是「記憶體安全」，其餘兩項作者已遺忘。
- 最驚人的宣稱是：Rust 將透過純靜態分析實現完全安全的記憶體管理，且完全無需垃圾回收機制。
- 作者離場時心想：「這不過是 Mozilla 的炒作噱頭。他們根本不可能用它開發瀏覽器，最終會像其他研究專案一樣無疾而終。」（他特別提及 DrServ/DrJS 作為 Mozilla 另一個失敗的研究專案）

#### 後續發展
- 其中一位同學後來攻讀博士，開展「通用型」GPU 語言專案，聲稱能在 GPU 上實現樹狀結構與圖形運算。作者認為此專案注定失敗，因 GPU 本質專注於資料並行運算，而非隨意的指標密集型結構。該專案最終確未實用化，但該同學仍取得博士學位，現於某科技巨頭參與 Rust 編譯器開發。

#### 作者對記憶體管理的探索歷程
- 作者曾對「無需 GC 的完全靜態記憶體安全」理念深深著迷（正是 Rust 最初的願景）。
- 他投入大量時間設計記憶體模型與靜態分析方案，試圖實現此理想。
- 某日他向指導教授 Kent Dybvig（傳奇語言 Chez Scheme 的作者）闡述想法時，Kent 平靜回應：  
  「完全的靜態記憶體管理——這真的可行嗎？記憶體管理本質上是動態過程。」
- 這句話擊碎了作者的幻想。他意識到精確的垃圾回收在通用情境下不可判定（與停機問題相關）。
- 當他改提參考計數法時，Kent 指出該方法開銷高昂，且效能常遜於優化的分代式 GC。若回收器設計得當，GC 暫停時間並非真正問題（Chez Scheme 即為明證）。

#### Chez Scheme 的對照啟示
- 作者對 Kent Dybvig 與 Chez Scheme 推崇備至：
  - 閃電級編譯速度
  - 可精細調校的低延遲 GC
  - 核心哲學：與其耗時優化拙劣程式碼，不如相信程式員能力，選擇簡潔適切的抽象方案
- 換言之，智慧判斷勝過粗暴的複雜設計。

#### Rust 的現實演變
- 「純靜態記憶體管理，徹底棄用 GC」的原始夢想已然破滅。
- 現代 Rust 實際包含：
  - `Rc<T>` / `Arc<T>`（透過 `Weak` 實現循環回收的參考計數）
  - `unsafe` 程式碼（網路協定堆疊、瀏覽器、作業系統核心等現實場景的必要手段）
  - 持續進行的安全機制研究（Stacked Borrows、Tree Borrows 等），但每個新模型都伴隨額外限制
- 然而官方宣傳與社群仍反覆強調「強靜態保證」與「無畏並行」。
- 當有人指出漏洞時，標準回應就變成「這是 Rust 的設計哲學」或「你未理解借用檢查器」。

#### 作者現今立場
- 他完全理解生命週期、借用等概念，甚至教授過進階 Rust 課程，指導學生用 Rust 實作函數式語言直譯器。
- 但越是深入教學，他越發欣賞 C++：
  - 對多數任務而言，C++ 的智慧指標與 RAII 更簡潔且表達力更強
  - 追求極致效能時，他仍會選擇 C/C++
- 他認為 Rust 的錯誤訊息難以解析，且持續與借用檢查器搏鬥令人疲憊
- 所謂「哲學辯護」更像是對設計妥協的宗教式開脫

#### 核心結論
作者認為現代 Rust 的成功主要歸因於行銷策略與生態系慣性，而非證明其核心理念較過往數十年技術（區域系統、Cyclone、線性類型、高效能 GC 等）更具根本優勢。他斷言最初的宏願實屬天真，後續妥協無可避免，而社群面對質疑時總以「哲學」作為遮羞布。
