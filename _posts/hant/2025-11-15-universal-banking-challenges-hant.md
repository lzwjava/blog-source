### 建立通用銀行應用程式的挑戰

建立一個單一、通用的銀行應用程式，服務所有國家的用戶——就像 Apple Pay 或 iCloud 那樣，核心功能保持一致，但為合規而數據存儲因地區而異——在技術上是可行的，但在銀行業面臨重大障礙。與專注於支付（Apple Pay）或雲端存儲（iCloud）的科技服務不同，全方位服務的銀行應用程式處理的是敏感的金融產品，如帳戶、貸款、投資和轉帳，這些都受到嚴格監管。各國的監管規定差異極大，包括反洗錢（AML）、認識你的客戶（KYC）、數據隱私（例如歐洲的 GDPR、美國的 CCPA、新加坡的 PDPA）、貨幣管制和當地許可要求。例如，渣打銀行（SC）在超過 60 個市場營運，但維持著針對特定區域的應用程式（例如 SC China、SC Mobile Hong Kong、SC Mobile USA 的對等版本通過合作夥伴關係，以及針對馬來西亞、台灣、孟加拉等其他市場的版本），因為「一刀切」的方法可能導致不合規、罰款或營運停擺。

Apple Pay 通過與當地支付網絡和銀行整合，同時保持用戶數據分隔（例如通過區域數據中心），在全球取得成功，但它並不管理完整的銀行業務；它只是一個錢包層。iCloud 同樣使用地理圍欄數據存儲來遵守像中國《網絡安全法》這樣的法律，但銀行業務涉及實時交易監管和向當地主管機關報告，這並非總能被抽象化。一個通用應用程式將需要動態功能開關（例如在某些區域啟用加密貨幣，在其他區域則封鎖）和後端路由至合規的數據中心，但即便如此，應用商店和監管機構可能仍會要求每個國家有單獨的列表或認證。

### 類似 GitHub Enterprise 的區域特定部署

如果完全通用的應用程式不可行，那麼一個以 GitHub Enterprise 為靈感的模式——將相同的核心平台進行區域部署並進行最少的自定義——對銀行來說更為合理。GitHub 提供具有數據駐留選項的 Enterprise Cloud（例如，為遵守 GDPR 將數據存儲在歐盟）或用於嚴格監管需求的本地伺服器，讓組織能夠在使用相同功能的同時滿足當地數據主權規則。銀行可以採用類似的「核心 + 區域覆蓋層」架構：

- **核心程式碼庫**：使用微服務建構模組化應用程式，共享組件（例如 UI/UX、交易處理引擎）在全球重複使用。
- **區域實例**：部署像「SC Mobile HK」或「SC Mobile SG」這樣的實例，每個實例託管在合規的數據中心（例如，針對新加坡/香港的 AWS 亞洲區域）。自定義將僅限於區域特定功能，例如與當地支付網關整合（例如香港的 FPS、新加坡的 PayNow）或適應稅務申報。
- **好處**：通過維護一個程式碼庫來減少重複開發，並使用 CI/CD 流水線進行自動化區域構建。像容器化（Docker/Kubernetes）這樣的工具能夠實現快速啟動，類似於 GitHub 處理企業部署的方式。

這種方法在金融科技中已有部分使用；例如，一些銀行使用來自像 Temenos 或 Backbase 這樣的供應商的白標平台，並針對每個市場進行自定義。然而，銀行仍然必須處理獨特的整合，例如連接國家身份證系統或中央銀行 API，這是 GitHub 不會面臨的。

### 銀行如何借鑒 Stripe 以減少重複工作

Stripe 展示了如何以更少的冗餘進行全球擴張：它提供統一的支付 API，在後台處理合規、欺詐檢測和貨幣轉換，同時針對當地法規進行優化。像渣打這樣的銀行可以吸取經驗以精簡營運：

- **統一 API 和模組化設計**：採用類似 Stripe 的 API 用於內部服務（例如，一個單一的支付 API，可路由到區域特定的處理器）。這最大限度地减少了每個國家的自定義代碼——專注於針對當地規則的「插件」，而不是重建所有東西。
- **自動化合規工具**：使用 AI 驅動的合規引擎（靈感來自 Stripe 用於欺詐檢測的 Radar），根據用戶位置自動應用 KYC/AML 檢查。Stripe 的全球收單業務能最優化地跨國路由交易；銀行可以與金融科技公司合作，實現類似的跨境流動，减少人工監管。
- **多貨幣和數據駐留**：通過預設使用當地貨幣並在區域存儲數據，來鏡像 Stripe 的多貨幣帳戶。這減少了資金管理中的重複工作。
- **擴展基礎設施**：Stripe 投資於全球基礎設施（例如，多個區域的數據中心）以實現無縫市場進入。銀行可以整合供應商生態系統（例如，一個具有區域合規認證的雲端供應商）並使用低代碼平台快速原型化功能，避免每個國家形成孤立的團隊。
- **成果**：更少的重複工作意味著更快的推廣速度——Stripe 在幾個月內進入新市場，而傳統銀行則需要數年。對於渣打而言，這可以將其應用程式統一在共享後端之下，可能節省開發成本（行業估計表明，模組化架構可帶來 30-50% 的效率提升）。

實際上，銀行正朝著這個方向發展：許多銀行正在採用「銀行即服務」（BaaS）模式或與 Stripe 本身合作以實現嵌入式金融。然而，與靈活的金融科技公司相比，遺留系統和更嚴格的監管（例如來自美國聯準會或新加坡金管局）拖慢了進度。
