## 1. 原始檔案基礎
- **檔案命名**：
  - 原始檔案以其包含的頂層類別之區分大小寫名稱命名
  - 檔案必須使用 UTF-8 編碼
  - 不建議在檔案名稱中使用特殊字符

## 2. 原始檔案結構
### 原始檔案組織
1. 套件宣告
2. 導入陳述式
3. 僅包含一個頂層類別

### 導入陳述式規則
- 禁止使用萬用字元導入
- 允許靜態導入
- 導入陳述式按特定順序組織：
  - 所有靜態導入
  - 所有非靜態導入
  - 每個群組內的導入按字母順序排列

## 3. 格式指引

### 縮排與大括號
- 使用 2 個空格進行縮排（不使用 Tab 鍵）
- 即使單行區塊也需在 `if`、`else`、`for`、`do` 和 `while` 陳述式中使用大括號
- 建議採用 K&R 風格的大括號放置方式
  ```java
  public void method() {
    if (condition) {
      // 程式碼區塊
    }
  }
  ```

### 行長度與換行
- 最大行長度為 100 個字符
- 優先在高層級語法斷點處換行
- 中斷方法呼叫鏈時，應在 `.` 之前進行換行

## 4. 命名慣例

### 通用規則
- **套件**：全小寫，不使用底線
- **類別**：UpperCamelCase
- **方法**：lowerCamelCase
- **常數**：UPPER_SNAKE_CASE
- **非常數字段**：lowerCamelCase
- **參數**：lowerCamelCase
- **區域變數**：lowerCamelCase

### 特定命名實踐
- 避免使用縮寫
- 例外名稱必須以 `Exception` 結尾
- 測試類別命名為 `TestClassName`

## 5. 程式設計實踐

### Java 語言規則
- **例外處理**：
  - 捕獲特定例外
  - 避免空的 catch 區塊
  - 始終包含詳細的錯誤訊息
- **Final 關鍵字**：
  - 對方法參數使用 `final`
  - 優先使用不可變物件
- **註解**：
  - 覆寫方法時必須使用 `@Override`
  - 適當使用標準註解

### 程式碼結構
- 優先使用組合而非繼承
- 保持方法簡短且專注
- 每行僅包含一個陳述式
- 避免深層嵌套的條件判斷

## 6. 註解與文件

### Javadoc 指引
- 所有公開類別和方法必須具備 Javadoc
- Javadoc 註解以 `/**` 開頭
- 包含描述、參數、回傳值和例外
- 使用 `@param`、`@return`、`@throws` 標籤

### 行內註解
- 謹慎使用
- 解釋「原因」而非「內容」
- 保持註解簡潔且有意義

## 7. 版本控制考量
- 避免提交被註解掉的程式碼
- 使用版本控制系統功能而非程式碼註解來追蹤變更

## 8. 效能考量
- 優先使用增強型 `for` 迴圈
- 字串連接時使用 `StringBuilder`
- 使用 try-with-resources 關閉資源
- 避免建立不必要的物件

## 9. 測試原則
- 編寫單元測試
- 追求高程式碼覆蓋率
- 使用 JUnit 進行測試
- 模擬外部依賴
- 測試邊界情況和錯誤條件

## 10. 附加工具
- Google 提供 Java 格式化工具和靜態分析工具
- 可使用 CheckStyle 和 Error Prone 來強制執行風格指引

## 結論
Google Java 風格指南提供全面的指引集，確保跨專案的 Java 程式碼保持一致性、可讀性和可維護性。
