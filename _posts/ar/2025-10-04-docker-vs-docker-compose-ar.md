### Docker مقابل Docker Compose: الاختلافات الرئيسية

Docker و Docker Compose مرتبطان ولكنهما يخدمان أغراضًا مختلفة في سير عمل containerization. Docker هي التكنولوجيا الأساسية، بينما يُعتبر Docker Compose أداة مبنية عليها للإعدادات الأكثر تعقيدًا.

#### ما هو Docker؟
Docker هو منصة لتطوير وشحن وتشغيل التطبيقات داخل **حاويات** خفيفة الوزن وقابلة للنقل. تقوم هذه الحاويات بتغليف التطبيق مع جميع تبعياته (الكود، وقت التشغيل، المكتبات، إلخ)، مما يضمن تشغيله بشكل متسق عبر بيئات مختلفة مثل التطوير والاختبار والإنتاج. تتفاعل مع Docker بشكل أساسي من خلال واجهة سطر الأوامر (CLI) باستخدام أوامر مثل `docker run`، و `docker build`، و `docker ps` لإدارة الحاويات الفردية.

#### ما هو Docker Compose؟
Docker Compose هو أداة orchestration تمد Docker للتعامل مع **تطبيقات multi-container**. يستخدم ملف YAML بسيط (عادةً `docker-compose.yml`) لتعريف كومة التطبيق بالكامل — بما في ذلك الخدمات المتعددة، والشبكات، والمجلدات volumes، ومتغيرات البيئة. بدلاً من التعامل مع العشرات من أوامر `docker run`، يمكنك تشغيل كل شيء بأمر واحد `docker-compose up`.

#### الاختلافات الرئيسية
إليك مقارنة سريعة:

| الجانب              | Docker                              | Docker Compose                          |
|---------------------|-------------------------------------|-----------------------------------------|
| **التركيز الأساسي**   | بناء وتشغيل وإدارة **الحاويات الفردية** | تعريف وتنسيق **تطبيقات multi-container** |
| **التكوين**   | وسائط سطر أوامر مضمنة (مثل `docker run -p 80:80 image`) | ملف YAML للإعداد التصريحي (الخدمات، المنافذ، المجلدات volumes) |
| **الأوامر**        | `docker run`، `docker build`، إلخ. | `docker-compose up`، `down`، `scale`، إلخ. |
| **النطاق**           | دورة حياة الحاوية منخفضة المستوى       | كومة تطبيقات عالية المستوى (مثل تطبيق + قاعدة بيانات + ذاكرة تخزين مؤقت cache) |
| **الشبكات / التبعيات** | إعداد يدوي لكل حاوية          | تلقائي (مثلاً، يمكن للخدمات الإشارة إلى بعضها البعض بالاسم) |
| **حالة الاستخدام**        | خدمات بسيطة ومعزولة           | تطبيقات معقدة مثل خوادم الويب مع قواعد البيانات |

باختصار: Docker يشبه مركبة ذات محرك واحد لمهمة واحدة، بينما Docker Compose هو مدير أسطول يقوم بتنسيق مركبات متعددة لرحلة واحدة.

### لماذا نحتاج إلى Docker Compose؟
بينما تعتبر Docker قوية بحد ذاتها لتطبيقات الحاوية الواحدة، فإن التطبيقات في العالم الحقيقي غالبًا ما تتضمن مكونات متعددة مترابطة (مثل واجهة أمامية frontend، وواجهة برمجة تطبيقات للخلفية backend API، وقاعدة بيانات، وذاكرة تخزين مؤقت Redis). إليك الأسباب التي تجعل Docker Compose أساسيًا:

- **يبسط إدارة الحاويات المتعددة**: بدونه، ستبدأ الحاويات وتوقفها يدويًا، وتقوم بتكوين الشبكات، والتعامل مع التبعيات عبر أوامر CLI متكررة — وهو ما يصبح فوضويًا وعرضة للأخطاء. يقوم Compose بأتمتة هذا بملف وأمر واحد.

- **القابلية للتكرار والتعاون**: يعمل ملف YAML كمخطط. يمكن لأعضاء الفريق تشغيل نفس البيئة تمامًا باستخدام `docker-compose up`، مما يجعل التطوير متسقًا وقابلًا للنقل (رائع لمسارات CI/CD).

- **الكفاءة في التطوير**: ميزات مثل ربط الخدمات (مثلاً، يتصل تطبيقك تلقائيًا بقاعدة البيانات عبر اسم المضيف hostname) وربط المجلدات volume لإعادة تحميل الكود المباشر تسرع عملية التكرار. يمكنك أيضًا قياس الخدمات (مثلاً، `docker-compose scale web=3`) دون إعادة بناء الصور.

- **عزل البيئة**: يقوم بإنشاء شبكات ومجلدات volumes معزولة لكل مشروع، مما يتجنب التعارضات عند تشغيل تطبيقات متعددة محليًا.

في جوهر الأمر، يحول Docker Compose أداة Docker من أداة فردية إلى لاعب جماعي لهندسات microservices الحديثة. إذا كنت قد بدأت للتو، جرب ملف `docker-compose.yml` بسيط لكومة WordPress + MySQL لترى السحر.
