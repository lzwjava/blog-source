Azure DevOps هي مجموعة شاملة من أدوات التطوير المستندة إلى السحابة والمقدمة من Microsoft لمساعدة الفرق على تخطيط وتطوير واختبار وتسليم البرامج بكفاءة. تندمج بسلاسة مع خدمات Microsoft الأخرى مثل Azure وGitHub، وتدعم منهجيات متنوعة مثل Agile وScrum وممارسات DevOps. يغطي هذا الدليل الأساسيات: ما هو، المكونات الرئيسية، كيفية البدء، أفضل الممارسات، والموارد للتعلم المتعمق.

## ما هو Azure DevOps؟
يوفر Azure DevOps قدرات DevOps شاملة، مما يتيح التعاون عبر التطوير والعمليات وأصحاب المصلحة. إنه محايد للأنظمة الأساسية، ويدعم لغات وأطر عمل وأدوات متعددة. تشمل المزايا الرئيسية:
- **القابلية للتوسع**: يتعامل مع مشاريع بأي حجم، من الفرق الصغيرة إلى المؤسسات الكبيرة.
- **الدمج**: يتصل مع بيئات التطوير المتكاملة مثل Visual Studio وGitHub وSlack وJira.
- **الأمان**: ميزات امتثال مدمجة مثل التحكم في الوصول المستند إلى الأدوار (RBAC) وسجلات التدقيق.
- **التسعير**: مجاني لما يصل إلى 5 مستخدمين؛ تبدأ الخطط المدفوعة من 6 دولارات لكل مستخدم/شهر للحصول على ميزات إضافية.

اعتبارًا من عام 2025، تطور Azure DevOps مع تحسينات في تكاملات الذكاء الاصطناعي (مثل GitHub Copilot for Azure) وتحليلات خطوط الأنابيب المحسنة.

## المكونات الرئيسية
يتكون Azure DevOps من خمس خدمات أساسية، يمكن الوصول إلى كل منها عبر بوابة ويب أو واجهات برمجة التطبيقات (APIs):

### 1. **لوحات العمل (Boards)**
   - **الغرض**: أدوات التخطيط والتتبع المرئية لعناصر العمل.
   - **الميزات**:
     - لوحات Kanban لتصور سير العمل.
     - قوائم المهام المتراكمة (Backlogs) لتحديد أولويات المهام.
     - سباقات التطوير (Sprints) للتكرارات الرشيقة (Agile).
     - استعلامات (Queries) لإعداد التقارير المخصصة.
   - **حالة الاستخدام**: تتبع الأخطاء والميزات والمهام في الوقت الفعلي.

### 2. **المستودعات (Repos)**
   - **الغرض**: التحكم في إصدارات الكود المركزية.
   - **الميزات**:
     - مستودعات Git أو TFVC.
     - استراتيجيات التفرع (Branching) وطلبات السحب (Pull Requests).
     - دمج Wiki للتوثيق.
   - **حالة الاستخدام**: التعاون في مراجعات الكود والحفاظ على السجل.

### 3. **خطوط الأنابيب (Pipelines)**
   - **الغرض**: أتمتة التكامل المستمر/النشر المستمر (CI/CD).
   - **الميزات**:
     - خطوط أنابيب قائمة على YAML أو كلاسيكية.
     - عمليات بناء واختبارات ونشر متعددة المراحل.
     - التكامل مع Azure Artifacts لإدارة الحزم.
     - بيئات (Environments) للموافقات والبوابات.
   - **حالة الاستخدام**: أتمتة عمليات البناء لكل commit والنشر إلى السحابة أو محليًا.

### 4. **خطط الاختبار (Test Plans)**
   - **الغرض**: الاختبار اليدوي والاستكشافي.
   - **الميزات**:
     - إدارة حالات الاختبار.
     - السجلات الحية (Live logs) والمرفقات.
     - التكامل مع الاختبارات المؤتمتة من Pipelines.
   - **حالة الاستخدام**: ضمان الجودة قبل الإصدار.

### 5. **القطع الأثرية (Artifacts)**
   - **الغرض**: إدارة الحزم والتعامل مع التبعيات.
   - **الميزات**:
     - حزم عالمية (Universal packages) ومخازن NuGet وnpm وMaven.
     - سياسات الاحتفاظ (Retention policies) للملفات الثنائية.
   - **حالة الاستخدام**: مشاركة وإصدار المكتبات عبر الفرق.

## البدء
اتبع هذه الخطوات لإعداد Azure DevOps:

1. **إنشاء حساب**:
   - انتقل إلى [dev.azure.com](https://dev.azure.com) وسجل الدخول باستخدام حساب Microsoft (هناك طبقة مجانية متاحة).
   - أنشئ مؤسسة جديدة (على سبيل المثال، "MyProjectOrg").

2. **إعداد مشروع**:
   - داخل مؤسستك، انقر على "New Project".
   - اختر Visibility (خاص/عام) و Version Control (Git/TFVC).
   - أضف أعضاء الفريق عبر دعوات البريد الإلكتروني.

3. **تكوين المستودعات (Repos)**:
   - استنسخ المستودع الافتراضي: `git clone https://dev.azure.com/{org}/{project}/_git/{repo}`.
   - ادفع الكود الأولي: `git add . && git commit -m "Initial commit" && git push`.

4. **بناء خط أنابيب بسيط**:
   - في Pipelines > New Pipeline > Select repo > ASP.NET (أو إطار العمل الخاص بك).
   - استخدم YAML للبساطة:
     ```yaml
     trigger:
     - main
     pool:
       vmImage: 'ubuntu-latest'
     steps:
     - task: DotNetCoreCLI@2
       inputs:
         command: 'build'
         projects: '**/*.csproj'
     ```
   - احفظ وشغل خط الأنابيب.

5. **إنشاء لوحة عمل (Board)**:
   - انتقل إلى Boards > Sprints > New Query.
   - حدد أنواع عناصر العمل (مثل Epic > Feature > Task).

6. **الاختبار والنشر**:
   - أضف مهمة اختبار (test task) في خط الأنابيب الخاص بك.
   - أنشئ خط أنابيب إصدار (release pipeline) للنشر إلى Azure App Service.

للحصول على دروس عملية، ابدأ مع البدايات السريعة الرسمية.

## أفضل الممارسات
- **اعتماد خطوط أنابيب YAML**: فهي خاضعة لتحكم الإصدار وقابلة لإعادة الاستخدام.
- **استخدم سياسات الفروع (Branch Policies)**: اشترط الحصول على موافقات لطلبات السحب وربط عناصر العمل.
- **نفّذ فحوصات الأمان**: ادمج أدوات مثل SonarQube أو Microsoft Defender.
- **راقب باستخدام التحليلات**: استخدم لوحات التحكم (Dashboards) لمعدل الإنجاز (velocity) ومعدلات نجاح البناء.
- **التوسع باستخدام الامتدادات**: يحتوي Marketplace على أكثر من 2000 امتداد (مثل إشعارات Slack).
- **الدعم الهجين**: للاستخدام محليًا، استخدم Azure DevOps Server (مستضاف ذاتيًا).

المزالق الشائعة: التخصيص المفرط في وقت مبكر — ابدأ ببساطة وقم بالتطوير التكراري.

## مواضيع متقدمة
- **الامتدادات والسوق (Extensions and Marketplace)**: خصص باستخدام الإضافات (plugins) لأدوات مثل Jenkins أو Terraform.
- **تكامل الذكاء الاصطناعي/التعلم الآلي (AI/ML Integration)**: استخدم Azure ML لنشر النماذج في خطوط الأنابيب.
- **التعاون متعدد الفرق**: استفد من المناطق (Areas) والتكرارات (Iterations) للمؤسسات الكبيرة.
- **الترحيل (Migration)**: أدوات لاستيراد البيانات من GitHub أو Jira.
