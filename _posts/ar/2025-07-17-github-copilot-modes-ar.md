### نظرة عامة
يقدم GitHub Copilot عدة أوضاع للمساعدة في كتابة التعليمات البرمجية في بيئات التطوير المتكاملة مثل VS Code وVisual Studio، بما في ذلك وضع التحرير (الذي يتماشى مع وصفك لـ"وضع الإصلاح" لتحديد وإصلاح أسطر أو دوال محددة) ووضع الوكيل (الذي يتعامل مع التعديلات الأوسع والأكثر استقلالية عبر الملفات أو المشاريع بأكملها). وضع التحرير دقيق ويخضع لسيطرة المستخدم، مما يجعله مثاليًا للإصلاحات المستهدفة، بينما يعمل وضع الوكيل كمبرمج زوجي ذكي، حيث يستنتج المهام المعقدة ويقوم بإجراء تغييرات عبر ملفات متعددة دون الحاجة إلى مدخلات مستمرة.[1][1] كلا الوضعين يعززان الإنتاجية ولكنهما يختلفان في النطاق والاستقلالية وسير العمل.

### الاختلافات الرئيسية
يركز وضع التحرير على مقاطع التعليمات البرمجية المحددة من قبل المستخدم، ويقدم اقتراحات للمراجعة والموافقة قبل تطبيق التغييرات. في المقابل، يعمل وضع الوكيل على مستوى أعلى، حيث يحلل سياق قاعدة التعليمات البرمجية بالكامل للتخطيط وتنفيذ والتكرار في التعديلات بشكل مستقل، وغالبًا ما يقوم بتعديل ملفات كاملة أو مكونات مرتبطة للحفاظ على الاتساق.[2][1] إليك مقارنة جنبًا إلى جنب:

| الميزة                  | وضع التحرير (وضع الإصلاح)                                                                | وضع الوكيل                                                              |
|--------------------------|--------------------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **النطاق**               | يقتصر على الأسطر أو الدوال المحددة، أو ملف واحد. تقوم بتحديد التعليمات البرمجية لإصلاح الأخطاء أو إعادة الهيكلة أو تحسين أجزاء محددة.[1] | مساحة العمل أو المشروع بأكمله. يقوم بتحديد وتحرير الملفات المرتبطة تلقائيًا، بما يتجاوز ما تحدده.[2][3] |
| **التحكم للمستخدم**      | عالي: يقترح تغييرات لمراجعتها وموافقتك الصريحة. أنت تحدد بالضبط ما يجب تحريره.[4] | متوسط: يطبق التعديلات تلقائيًا لكنه يعلّم الأوامار عالية المخاطر (مثل تشغيل الطرفية) للمراجعة. أنت تحدد الهدف عبر أوامر باللغة الطبيعية.[1][1] |
| **الاستقلالية**          | منخفضة: يقدم اقتراحات مستهدفة؛ لا يستنتج عبر الملفات أو ينفذ إجراءات مستقلة.[1] | عالية: يستنتج خطوة بخطوة، يشغّل الاختبارات/الأوامر، يكتشف الأخطاء ويصححها ذاتيًا. يحافظ على السياق عبر الجلسات.[2][3] |
| **وقت الاستجابة**        | سريع: تحليل سريع للتحديد فقط.[2] | أبطأ: يحلل سياق المشروع الكامل، مما قد يستغرق وقتًا أطول لقواعد التعليمات البرمجية الكبيرة.[2] |
| **الأفضل لـ**            | الإصلاحات السريعة مثل تصحيح خطأ في دالة، تحسين حلقة، أو إعادة كتابة طريقة دون تأثير أوسع.[1] | المهام المعقدة مثل إعادة الهيكلة عبر الملفات، إنشاء اختبارات لوحدة نمطية، ترحيل التعليمات البرمجية، أو بناء ميزات من الصفر.[3][5] |
| **أمثلة**                | - حدد دالة تحتوي على خطأ: "أصلح فحص القيمة null هذا."<br>- سلط الضوء على أسطر: "اجعل هذا غير متزامن." [2] | - أمر: "أعد هيكلة طبقة الخدمة بأكملها لاستخدام async/await وقم بتحديث جميع التبعيات."<br>- أو: "حدث مشروع Java هذا إلى JDK 21 عبر جميع الملفات." [5][6] |
| **المخاطر/القيود**       | مخاطر ضئيلة، حيث أن التغييرات معزولة؛ لكنها تتطلب تحديدًا يدويًا لكل إصلاح.[1] | الاستقلالية العالية يمكن أن تؤدي إلى تغييرات غير مقصودة؛ راجِع دائمًا الاختلافات. ليس مثاليًا للبيئات ذات التحكم العالي.[7][4] |

### حالات الاستخدام وسير العمل
- **وضع التحرير للإصلاحات المستهدفة**: استخدم هذا عندما تعرف بالضبط ما الخطأ، على سبيل المثال، تحديد التعليمات البرمجية المعرضة للخطأ في دالة لحل خطأ أو تحسين الأداء. إنها أشبه بأداة "تحرير موضعي" — حدد التعليمات البرمجية في بيئة التطوير الخاصة بك، وأمر Copilot عبر الدردشة (مثل "@workspace /fix")، وقم بتطبيق معاينة الاختلاف. يبرز هذا الوضع في التطوير التكراري حيث تريد الحفاظ على التحكم الكامل وتجنب إصلاح المناطق غير المتأثرة. على سبيل المثال، في مشروع .NET، قد تحدد طريقة وتطلب، "حدد استثناءات المرجعية الفارغة واقترح إصلاحات" لتلك المقتطفات فقط.[2][8] وهو متاح في VS Code وVisual Studio مع إضافات GitHub Copilot.

- **وضع الوكيل للتعديلات على مستوى المشروع**: فعّل هذا للتغييرات الشاملة، مثل عندما تحتاج إلى تحرير ملفات كاملة أو تنسيق التحديثات عبر قاعدة التعليمات البرمجية. ابدأ جلسة في Copilot Chat (مثل "#agentmode" أو عبر القائمة المنسدلة)، وأعط أمرًا عالي المستوى مثل "ابحث عن جميع استخدامات واجهة برمجة التطبيقات القديمة وقم بترحيلها إلى الجديدة في هذا المشروع"، وشاهد وهو يخطط للخطوات: تحليل الملفات، اقتراح التعديلات، تشغيل الاختبارات، والتكرار. يمكنه إنشاء ملفات جديدة، تحديث مساحات الأسماء، أو حتى إنشاء أقسام من تطبيق. في تحديث Java، على سبيل المثال، يقوم بمسح مشروع قديم، وتحديث تبعيات Gradle، والتحقق من صحة التغييرات عبر ملفات متعددة.[5][3] هذا الوضع قوي بشكل خاص لإعادة الهيكلة، البحث عن الأخطاء على نطاق واسع، أو أتمتة المهام المتكررة مثل إضافة التوثيق أو الاختبارات.[6][9]

يبني وضع الوكيل على أساس وضع التحرير لكنه يوسعه — فكر في وضع التحرير كمشرط للقطع الدقيق، والوكيل كجراح يتعامل مع العملية الكاملة.[1] يمكن للتوجيهات المخصصة (عبر إعدادات VS Code على سبيل المثال) توجيه وضع الوكيل للاتساق، مثل فرض اصطلاحات التسمية عبر التعديلات.[1]

### متى تختار كل وضع
- اختر **وضع التحرير/الإصلاح** إذا كانت مهمتك موضعية (مثل إصلاح دالة واحدة) للحفاظ على البساطة والسرعة.
- انتقل إلى **وضع الوكيل** للكفاءة في النطاقات الأكبر، مثل تحرير الملف بالكامل أو إعادة الهيكلة متعددة الخطوات، لكن ابدأ بمهام صغيرة لبناء الثقة في مخرجاته.[2][1] كلاهما يتطلب اشتراك GitHub Copilot (Pro أو أعلى) ومدمجان في VS Code/Visual Studio. للحصول على أفضل النتائج، اجمع بينهما: استخدم وضع التحرير للإصلاحات الأولية، ثم وضع الوكيل لنشر التغييرات على مستوى المشروع.[8] إذا كنت تتعامل مع تعليمات برمجية حساسة، راجع جميع مقترحات الوكيل بعناية لتجنب المفاجآت.[7]
